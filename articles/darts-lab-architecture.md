## Webアプリは「レストラン」に似ている

Webアプリの仕組みを理解するために、レストランに例えてみよう。

| レストラン | Webアプリ | 役割 |
|-----------|----------|------|
| ホール（客席） | フロントエンド | ユーザーが見る画面 |
| ウェイター | API | 注文（リクエスト）を厨房に伝える |
| 厨房 | バックエンド | リクエストを処理して結果を返す |
| 冷蔵庫・倉庫 | データベース | データを保管する場所 |
| 外部の仕入れ業者 | 外部サービス | 決済、通知など専門的な処理を担当 |

この記事では、67,000行のダーツスタッツ管理アプリ「darts Lab」の全体像を使って、Webアプリがどう動いているかを解説する。

## darts Lab の4層構成

darts Lab は大きく4つの層で構成されている。

```
① クライアント（ユーザーの画面）
    ↓ HTTPS通信
② Vercel（サーバーレス基盤）
    ↓
③ Firebase（データの保管庫）
    ↓
④ 外部サービス（Stripe / LINE / DARTSLIVE / Sentry）
```

### ① クライアント — ユーザーが触る画面

**Next.js 16** というフレームワークで作られている。フレームワークとは「アプリの骨組みを提供してくれるツール」だ。自分でゼロから作る必要がなく、ページの切り替え、画面の描画、サーバーとの通信といった基本機能が最初から揃っている。

darts Lab には30以上のページがあり、スタッツ分析画面だけで53個のコンポーネント（画面の部品）がある。

### ② Vercel — リクエストを処理するサーバー

**Vercel** は「サーバーレス」のホスティングサービスだ。

**サーバーレスとは？** 従来のサーバーは24時間365日動き続ける（＝家賃がかかる）。サーバーレスは「リクエストが来たときだけ起動する」仕組みだ。コンビニの無人レジのようなもので、お客さんが来たときだけ動き、いないときは電気代がかからない。

darts Lab では40本以上のAPIルート（サーバー側の処理）がVercel上で動いている。

### ③ Firebase — データの保管庫

**Firebase** はGoogleが提供するBaaS（Backend as a Service）。以下の3つを一括で提供してくれる。

- **Firebase Auth** — ログイン機能（メール/パスワード、Google認証）
- **Firestore** — NoSQLデータベース（23コレクション）
- **Storage** — 画像ファイルの保管

認証とデータベースが同じプラットフォームにあるので、「ログインしたユーザーだけが自分のデータを読める」というアクセス制御がシンプルに書ける。

### ④ 外部サービス — 専門業者に任せる

自分で作らず、専門のサービスに任せている部分がある。

| サービス | 役割 | なぜ外部に任せるか |
|---------|------|--------------------|
| **Stripe** | サブスク決済 | クレジットカード情報の管理は自前でやるべきではない |
| **LINE** | 通知・Bot | LINEのメッセージ送信はLINEのAPIを使うしかない |
| **DARTSLIVE** | スタッツ取得 | ダーツのプレイデータはDARTSLIVEが持っている |
| **Sentry** | エラー監視 | 本番でエラーが起きたら即座に通知してくれる |

## なぜこの構成を選んだか

理由は3つ。

**1. 月額ほぼ0円で運用できる。** Vercel、Firebase、Upstash Redis、Sentry——すべて無料枠で収まっている。唯一の固定費はドメイン代の約1,500円/年だけ。

**2. スケーラブル。** ユーザーが増えても、サーバーレスなら自動でスケールする。自分でサーバーの台数を増やす必要がない。

**3. 個人で運用できる。** サーバーの監視やメンテナンスが不要。フリーランスの案件と並行しながら一人で開発・運用できる構成だ。

## まとめ

Webアプリは「フロント → サーバー → DB → 外部サービス」の層構造で動いている。darts Lab では Next.js + Vercel + Firebase + 外部サービスという組み合わせで、月額ほぼ0円・一人運用を実現した。
