## はじめに

ダーツ分析 Web アプリ「**darts Lab**」の 2026年3月第2週（3/6〜3/10）の開発記録です。
この週は**スタッツ画面の大規模リデザイン**、**デモアカウントの多層防御**、**全画面エラーハンドリング監査**など、プロダクトの成熟度を引き上げる作業に集中しました。

**darts Lab のスペック:**
- Next.js 16 + React 19 + MUI 7 + Firebase
- TypeScript **82,000行** / **41 API** / **144コンポーネント** / **636テスト**
- Claude Code によるAI駆動開発

---

## 3/6（木）— スタッツ画面を22カードから17カードに整理

この日だけで **44コミット**。最も密度の高い1日でした。

### やったこと

**スタッツページ再構成:**
- 情報過多だったスタッツ画面のカードを22個から17個に削減
- `CountUpDeepAnalysisCard`（1,598行の巨大コンポーネント）を5つのサブコンポーネントに分割
- SessionCompareCard を表形式UIにリデザインし、ヒートマップ・ワンブル率・ロートン率・ハットトリック率を統合

**LINE通知の画像リッチ化:**
- セッション比較画像を10メトリクスに拡張
- ミス方向ヒートマップ画像を追加
- Noto Sans JP フォントURLの形式修正、Storage bucket名の末尾改行トリムなど、実際にLINE APIへ送信して初めて発覚する問題を10件以上修正

**新機能の種まき:**
- HealthKit連携（Capacitor 8対応）
- アフィリエイトクリック追跡API
- Playwright E2Eテスト基盤

### 課題・気づき

スキルレーダーの指標名を何度も変更しました（HT率→ハット率→ハットトリック率）。ダーツ指標の命名規則をスタイルガイドとして事前に固めておくと、UIラベルの手戻りを防げます。

LINE通知の画像送信は、ローカルでは再現しにくいエッジケースが多いです。フォントURL・画像パス・バケット名の改行など、本番デプロイして初めて発覚する問題ばかりでした。

## 3/7（金）— デモアカウント導入 & HealthKit 11機能実装

### やったこと

**デモアカウントシステム:**
- 3段階のデモアカウント（General / Pro / Admin）を導入
- ポートフォリオからワンクリックでサービスを体験できる導線を構築
- README をポートフォリオ水準にリッチ化

**分析機能の拡充:**
- COUNT-UP × ヘルスケアの相関分析
- 練習時の「意識ポイント」機能（最大3つ設定可能）
- 目標の月次リセット機能

**HealthKit連携（Capacitor 8）:**
- 11機能を実装（歩数・心拍数・睡眠時間などの取得+ダッシュボード表示）
- Capacitor 8 のプラグイン登録で `registerPlugin` の呼び出し方に苦戦
- 画面上にデバッグログを表示する仕組みを入れてボトルネックを特定

### 課題・気づき

Capacitor のネイティブプラグイン呼び出しは、`console.log` がネイティブ側で見えないケースがあります。画面上に直接デバッグログを出す手法が有効でした。

## 3/8（土）— 全画面監査 & CI安定化

### やったこと

**品質改善:**
- 全画面を対象にエラーハンドリングの監査を実施
- 大きすぎるページコンポーネントを分割
- ダークモードのCSS上書き問題を修正（プロフィール画像のセッション伝播含む）

**CI/スクレイピング修正:**
- バレルスクレイピングのタイムアウトを50分に延長
- Puppeteer の sandbox 設定を CI 環境向けに修正
- Node 22 対応のキャッシュ修正

### 課題・気づき

CI環境とローカル環境の差異（Puppeteer + Node 22）に起因する問題が多発しました。ヘッドレスブラウザを使うスクレイピングは、環境ごとのサンドボックス設定・フォント・タイムアウトの違いに注意が必要です。

## 3/10（月）— デモアカウント強化 & 開発環境整備

### やったこと

**デモアカウントの多層防御:**
- 7種類のシードデータジェネレータを追加（ダーツ設定・目標・XP履歴・意識ポイント・ブックマーク・ディスカッション）
- `useDemoGuard` フックを14コンポーネントに適用（クライアントサイド制限）
- Firestore Security Rules に `isNotDemo()` ヘルパーを追加（サーバーサイド制限）
- API ルートのブロックリスト拡張

**開発環境の最適化:**
- SubAgents 構成の導入（PM → 実装 → テスト → レビュー → 日記の5段階フロー）
- 週次開発日記の自動生成 GitHub Actions ワークフロー
- 3リポ間のクロスリポ同期ルールを CLAUDE.md に明文化
- メトリクス自動抽出スクリプト（`npm run metrics`）

### 課題・気づき

デモアカウントの書き込み制限は「クライアントサイドのトースト → API ルートのブロック → Firestore ルールの拒否」の3層で実装しました。どれか1つが突破されても他の層で防げる多層防御のアプローチです。
