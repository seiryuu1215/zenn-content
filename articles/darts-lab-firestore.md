## データベースとは何か？

データベースは、アプリのデータを保存・管理する仕組みだ。Excelの超強力版だと思ってほしい。ユーザー情報、投稿データ、設定値——アプリが扱うすべてのデータがここに入る。

## RDB と NoSQL — 2つのデータベースの流派

データベースには大きく2つの種類がある。

| | RDB（リレーショナル） | NoSQL（ドキュメント型） |
|---|---|---|
| イメージ | Excel の表 | フォルダとファイル |
| データ構造 | 行と列で固定 | 自由な形式 |
| 関連データ | JOIN で結合 | ドキュメントに埋め込み |
| 変更 | マイグレーション（設計変更手続き）が必要 | フィールドを自由に追加できる |
| 代表例 | PostgreSQL, MySQL | Firestore, MongoDB |

darts Lab では **Firestore**（NoSQL）を選んだ。理由は、Firebase Auth との一体性と、スキーマレス（構造を柔軟に変えられる）な特性が個人開発に向いていたからだ。

## Firestore の基本概念

Firestore のデータ構造は「フォルダとファイル」で理解できる。

- **コレクション** = フォルダ（データのグループ）
- **ドキュメント** = ファイル（1件のデータ）
- **フィールド** = ファイルの中身（名前、年齢などの項目）
- **サブコレクション** = フォルダの中のフォルダ

## darts Lab の23コレクション

darts Lab は「12のルートコレクション + 11のサブコレクション = 計23コレクション」で構成されている。

```
Firestore Root（ルートフォルダ）
├── users/         ← ユーザー情報（11のサブコレクションを持つ）
├── darts/         ← セッティング投稿
├── barrels/       ← バレルDB（7,000件以上）
├── articles/      ← 記事
├── discussions/   ← 掲示板
├── comments/      ← コメント
└── ...（計12個）
```

`users` の下にはさらにサブコレクションがぶら下がっている。

```
users/{userId}/
├── dartsLiveStats/    ← スタッツ履歴
├── goals/             ← 目標
├── xpHistory/         ← XP獲得履歴
├── notifications/     ← 通知
└── ...（計11個）
```

### なぜサブコレクションにするのか？

理由は3つ。

**1. セキュリティが自然に書ける。** `users/{userId}/goals/{goalId}` というパスなら、パスを見るだけで「このデータの所有者は userId」とわかる。

**2. クエリが効率的。** `users/自分のID/goals` を読めば自分の目標だけが返る。全ユーザーのデータから絞り込む必要がない。

**3. 削除が簡単。** ユーザー退会時にサブコレクションごと消せる。

一方、`darts`（セッティング）や `articles`（記事）はルートコレクションにした。これらは **他のユーザーも閲覧する** データだから、特定ユーザーの下に置くと全ユーザーを横断して検索する必要が出てしまう。

## 非正規化 — 同じデータをわざとコピーする

Firestore には SQL の `JOIN`（テーブル同士を結合する機能）がない。そこで darts Lab では **非正規化** という戦略を取った。

```typescript
// darts コレクションのドキュメント
{
  userId: "abc123",
  userName: "青竜",           // ← users コレクションからコピー
  userAvatarUrl: "https://...", // ← users コレクションからコピー
  title: "トルピード系セッティング",
  barrel: { name: "RISING SUN" }
}
```

ユーザー名とアバターURLを各ドキュメントに埋め込んでいる。こうすることで、セッティング一覧を表示するとき **1回のクエリ** で完結する。

**トレードオフ**: ユーザーがプロフィールを更新したら、全コピーを書き換える必要がある。ただ、プロフィール変更は月に1回もない。一方、一覧表示は毎日何十回も発生する。読み取り頻度 >> 書き込み頻度なので、非正規化が合理的だった。

## セキュリティルール — データベースの最後の砦

Firestore にはセキュリティルールという仕組みがある。「誰が何を読み書きできるか」をデータベースレベルで制御する。

```javascript
// ユーザーが自分のプロフィールを更新するとき
// role や subscriptionId の書き換えは拒否する
allow update: if isOwner(userId)
  && !request.resource.data.diff(resource.data)
      .affectedKeys().hasAny(['role', 'subscriptionId']);
```

これにより、ブラウザの開発者ツールから `role: 'admin'` に書き換えるような攻撃をデータベースレベルでブロックしている。

## まとめ

- データベースには RDB と NoSQL の2種類がある
- Firestore は「フォルダとファイル」構造で柔軟にデータを管理
- サブコレクションでセキュリティとクエリ効率を両立
- 非正規化で読み取りパフォーマンスを最適化
- セキュリティルールでデータの安全性を担保
