## この記事の使い方

darts Lab（67,000行のダーツスタッツ管理アプリ）で使っている技術を、**用語 → 意味 → なぜこれを選んだか**の3点セットで解説する。シリーズの各記事で「この単語なんだっけ？」となったら、ここに戻ってくればOK。

## フロントエンド（ユーザーが見る画面）

| 用語 | 意味 | 補足：なぜこれを選んだか |
|------|------|--------------------------|
| **Next.js** | Reactベースのフレームワーク。ページの切り替え、サーバー側レンダリング、APIルートなどを一括で提供する | Nuxt.js（Vue系）やRemixも候補だったが、React のエコシステムが最大で情報量が多い。MUIなどのUIライブラリも React 向けが充実している |
| **React** | 画面を「コンポーネント」という部品に分けて組み立てるUIライブラリ | Vue.js やSvelteも人気だが、求人数・ライブラリ数ともに React が最多。学習コストはやや高いが長期的に有利 |
| **MUI（Material UI）** | Googleのデザインガイドライン「Material Design」に沿ったUIコンポーネント集 | Chakra UIやshadcn/uiも候補。MUIはコンポーネント数が豊富でテーマのカスタマイズ性が高い。ダークテーマ対応も標準搭載 |
| **コンポーネント** | 画面の再利用可能な部品。ボタン、カード、グラフなど | darts Lab ではスタッツ画面だけで53個のコンポーネントがある |
| **App Router** | Next.js のルーティング方式。ファイルを置くだけでページのURLが決まる | `app/stats/page.tsx` → `/stats` のように、フォルダ構造がそのままURLになる |
| **TypeScript** | JavaScriptに「型」を追加した言語。変数に入るデータの種類を事前に決められる | 型があるとエディタが間違いを教えてくれるので、バグが減る。大規模開発では事実上必須 |

## バックエンド・サーバー

| 用語 | 意味 | 補足：なぜこれを選んだか |
|------|------|--------------------------|
| **API（エーピーアイ）** | アプリの「注文窓口」。フロントエンドからのリクエストを受け取り、処理結果を返す | darts Lab には40本以上のAPIルートがある |
| **APIルート** | Next.js のサーバー側処理。`app/api/` フォルダにファイルを置くと、そのままAPIのURLになる | 別途バックエンドサーバーを立てる必要がなく、フロントとバックを一つのプロジェクトで管理できる |
| **REST API** | APIの設計スタイル。HTTPメソッド（GET/POST/PUT/DELETE）でデータの取得・作成・更新・削除を表現する | GraphQLという別のスタイルもあるが、シンプルさを優先してRESTを採用 |
| **ミドルウェア** | リクエストが本体の処理に届く前に実行される「関所」。認証チェックやレートリミットなどに使う | darts Lab では認証・エラーハンドリング・レートリミットの3つをミドルウェアで処理している |
| **サーバーレス** | リクエストが来たときだけサーバーが起動する仕組み。使わない時間は課金されない | 常時起動のVPS（Linuxサーバー）に比べて運用コストがほぼ0円。個人開発に最適 |
| **Webhook（ウェブフック）** | 外部サービスから「イベントが起きたよ」と通知を受け取る仕組み | Stripeの決済完了やLINEのメッセージ受信を、こちらから定期的に確認（ポーリング）する代わりに通知してもらう |
| **Cron（クロン）** | 決まった時刻に自動実行されるスケジュールタスク | darts Lab では毎朝10時にDARTSLIVEのスタッツ取得が走る。Vercel Cronを使用 |

## ホスティング・デプロイ

| 用語 | 意味 | 補足：なぜこれを選んだか |
|------|------|--------------------------|
| **Vercel（ヴァーセル）** | Next.js を作った会社が提供するホスティングサービス。デプロイ（公開）がワンクリック | AWS Amplify、Netlify、Cloudflare Pagesも候補。Next.js との相性が最高で、Hobby（無料）プランでも十分な機能がある |
| **デプロイ** | 開発したコードを本番サーバーに反映して、ユーザーが使える状態にすること | Vercel はGitHubにpushするだけで自動デプロイされる |
| **CI/CD** | コードの変更を自動でテスト（CI）し、問題なければ自動で公開（CD）する仕組み | GitHub Actions でlint→テスト→ビルドを自動実行。通ったらVercelがデプロイする |
| **GitHub Actions** | GitHub上でCI/CDパイプラインを構築するサービス | 無料枠が月2,000分あり、個人開発なら十分 |

## データベース

| 用語 | 意味 | 補足：なぜこれを選んだか |
|------|------|--------------------------|
| **Firestore** | Googleが提供するNoSQLデータベース。データを「コレクション（フォルダ）→ ドキュメント（ファイル）→ フィールド（中身）」の階層で管理する | PostgreSQLやMySQLなどのRDB（リレーショナルDB）も候補。Firestoreは①Firebase Authとの連携が簡単②セキュリティルールが書ける③無料枠が大きい（1日5万読み取り） |
| **NoSQL** | テーブルと行ではなく、JSON形式でデータを保存するデータベースの種類 | MongoDBも同じNoSQLだが、Firestoreはサーバー管理不要（フルマネージド）で運用が楽 |
| **RDB（リレーショナルDB）** | テーブル（表）同士を「リレーション（関係）」で結んでデータを管理するデータベース | PostgreSQL、MySQLが代表例。複雑な集計クエリが得意だが、サーバー管理が必要な場合が多い |
| **コレクション** | Firestoreのデータの入れ物。フォルダのようなもの | darts Lab では23コレクション（ルート12 + サブ11）を設計 |
| **ドキュメント** | コレクション内の1つのデータ。JSONオブジェクトのようなもの | 1ドキュメントの上限は1MB |
| **セキュリティルール** | Firestoreへの読み書き権限を制御するルール。「自分のデータだけ読み書き可」のような制限を宣言的に書ける | RDBだとサーバー側のコードで制御する必要があるが、Firestoreはルールファイルだけで制御できる |

## 認証・認可

| 用語 | 意味 | 補足：なぜこれを選んだか |
|------|------|--------------------------|
| **認証（Authentication）** | 「あなたは誰？」を確認すること。ログイン処理のこと | メール/パスワードとGoogle OAuth の2種類を提供 |
| **認可（Authorization）** | 「あなたは何ができる？」を判定すること。権限チェック | general / pro / admin の3段階ロール |
| **NextAuth.js** | Next.js 向けの認証ライブラリ。セッション管理やOAuth連携を簡単に実装できる | Auth0やClerkも候補。NextAuthはOSSで無料、Next.jsとの統合が深い。Firebase Authと組み合わせてデュアル認証を実現 |
| **Firebase Auth** | Firebaseの認証サービス。メール/パスワード、Google、Apple等のログインを提供 | NextAuthだけでも認証はできるが、Firestoreのセキュリティルールと連携するためにFirebase Authも併用（デュアル認証） |
| **JWT（ジョット）** | JSON Web Token。ログイン情報を暗号化したトークン。サーバーはこれを検証してユーザーを識別する | セッションをDBに保存する方式より軽量。Cookieに保存してリクエストごとに送信する |
| **OAuth（オーオース）** | 「Googleでログイン」のように、外部サービスの認証を借りる仕組み | ユーザーはパスワードを新たに設定する必要がなく、登録のハードルが下がる |
| **RBAC** | Role-Based Access Control。ユーザーの「役割（ロール）」に基づいてアクセス権を制御する仕組み | darts Lab では general（無料）/ pro（有料）/ admin の3ロールで機能を制限 |

## 決済

| 用語 | 意味 | 補足：なぜこれを選んだか |
|------|------|--------------------------|
| **Stripe（ストライプ）** | オンライン決済プラットフォーム。サブスクリプション（月額課金）機能を提供する | PayPal、Square、PAY.JPも候補。Stripeは①開発者向けドキュメントが最も充実②Webhookの仕組みが堅牢③テスト環境が本番と同じように使える |
| **Checkout** | Stripeが提供する決済ページ。カード情報の入力画面を自前で作る必要がない | PCI DSS（カード情報のセキュリティ基準）への対応をStripeに任せられる |
| **サブスクリプション** | 月額・年額の定期課金モデル | darts Lab ではProプランの月額課金にStripe Subscriptionsを使用 |
| **冪等性（べきとうせい）** | 同じ処理を2回実行しても結果が変わらない性質 | Webhookは通信障害で重複送信されることがあるため、同じイベントを2回処理しても二重課金にならないよう設計する |

## 外部サービス連携

| 用語 | 意味 | 補足：なぜこれを選んだか |
|------|------|--------------------------|
| **LINE Messaging API** | LINEのBot機能。メッセージ送信、プッシュ通知、リッチメニューなどを提供 | Slack BotやDiscord Botも候補だが、日本のユーザーはLINEの利用率が圧倒的に高い |
| **Flex Message** | LINEのカード形式のリッチメッセージ。HTMLのように自由にレイアウトを組める | テキストメッセージより情報量が多く、スタッツレポートの表示に最適 |
| **Puppeteer（パペティア）** | Googleが開発したヘッドレスブラウザ操作ツール。プログラムでWebページを操作できる | DARTSLIVEに公式APIがないため、ログインしてスクレイピングする必要がある。PlaywrightやSeleniumも候補だが、Puppeteerは軽量でVercelとの相性が良い |
| **スクレイピング** | Webサイトからプログラムでデータを取得すること | APIが提供されていないサービスからデータを取得する手段 |
| **Sentry（セントリー）** | エラー監視サービス。本番環境でエラーが起きたら自動で通知してくれる | Datadogも候補だが、Sentryは無料枠が大きく（月5,000イベント）、Next.jsとのSDK連携が簡単 |

## セキュリティ

| 用語 | 意味 | 補足：なぜこれを選んだか |
|------|------|--------------------------|
| **レートリミット** | 一定時間内のリクエスト数を制限する仕組み。DoS攻撃やAPIの濫用を防ぐ | darts Lab では60リクエスト/分/IPに制限。Upstash Redisで管理 |
| **Upstash Redis** | サーバーレス対応のRedis（インメモリDB）サービス | AWS ElastiCache等の通常のRedisはサーバー管理が必要。Upstashはサーバーレスで、REST API経由でアクセスできる。無料枠あり |
| **AES-256-GCM** | データの暗号化方式。鍵を知っている人だけがデータを復号できる | DARTSLIVEの認証情報をFirestoreに保存する際の暗号化に使用 |
| **署名検証** | Webhookが本当に正規のサービスから送られたかを暗号的に検証する仕組み | Stripeのシークレットキーで署名を検証し、なりすましを防ぐ |
| **多層防御** | セキュリティを1つの仕組みに頼らず、複数の層で守る考え方 | darts Lab では①レートリミット②認証③権限チェック④Firestoreルールの4層で防御 |

## アーキテクチャ用語

## まとめ

darts Lab では約30種類の技術・サービスを組み合わせている。選定の基準は3つ。

1. **無料枠があること** — 個人開発ではランニングコストが命取りになる
2. **ドキュメントが充実していること** — 詰まったときに自力で解決できる
3. **エコシステムが大きいこと** — ライブラリやツールが豊富で、組み合わせの幅が広い

「全部を一度に覚える必要はない。」シリーズの各記事で出てきた用語をこのページで確認しながら、少しずつ理解を深めていけばいい。
