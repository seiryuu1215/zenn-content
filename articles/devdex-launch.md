## なぜ作ったか

フリーランスで案件に入るたびに「この技術、ちゃんと説明できるか？」という不安があった。

スキルシートには書いてある。でも面談前日に「RLSってどう説明するんだっけ」と焦った経験が何度もある。Notionに書き溜めても続かない。入力が面倒だし、可視化されないから振り返る動機がない。

**「自分の技術理解度を、育てながら管理できるツール」が欲しかった。** それがDevDexの始まりだ。

## Claude Codeで「どう」作ったか

この記事の本題。DevDexの全コードは **Claude Code** で書いた。人間（自分）がやったのは**意思決定とフィードバック**だけ。

### 6体のサブエージェント体制

<!-- TODO: サブエージェント体制の図解画像 -->

Claude Codeの `.claude/agents/` に6体のエージェントを定義し、役割分担させた。

| エージェント | 役割 | やること |
|-------------|------|---------|
| **PM Agent** | 要件整理 | 要件をヒアリングしてIssueに分解、設計判断を記録 |
| **Implement Agent** | 実装 | コーディング・ファイル作成・修正 |
| **Test Agent** | テスト | TDDでテストを書き、全件パスを確認 |
| **Review Agent** | レビュー | セキュリティ・設計・品質の観点でチェック |
| **Diary Agent** | 日記 | PRごとに開発日記を自動生成 |
| **Biz Agent** | ビジネス | 課金設計・競合調査・KPI設計 |

**1つの機能追加の流れ:**

```
ユーザー「エンゲージメントアンケート機能が欲しい」
  ↓ PM Agent: 要件整理 → 4つの子Issueに分解
  ↓ Implement Agent: DB設計 → API → UI を順番に実装
  ↓ Test Agent: ユニットテスト32件を追加
  ↓ Review Agent: CRITICAL 2件を検出 → 修正
  ↓ PR作成 → マージ
```

人間が介入したのは最初の一言と、レビュー指摘の承認だけ。

### CLAUDE.mdが全ての起点

プロジェクトルートの `CLAUDE.md` に **設計方針・DB構造・コード規約・Git運用ルール** を全部書いた。これが6体のエージェント全員の共通認識になる。

```markdown
# CLAUDE.md の主な内容
- プロダクト概要とターゲット
- 技術スタック
- DBスキーマ（テーブル定義 + 設計意図）
- コミット規約（type: 日本語の説明）
- GitHub運用ルール（1 issue = 1 branch = 1 PR）
- SubAgentsフロー
```

**ポイント: 「なぜそう設計したか」も書く。** たとえば `relatedTermIds` を配列でなく中間テーブルにした理由まで書いておくと、Implement Agentが同じ設計思想で新機能を実装してくれる。

### プロンプト設計のコツ

一番効いたのは **「役割を分けること」**。

実装・テスト・レビューを同じ会話でやらせると品質が下がる。エージェントごとに「あなたはTest Agentです。実装はしない」と明示することで、出力が安定した。

うまくいかなかったのは曖昧な指示。「いい感じにして」は最悪で、「このAPIのレスポンスをこの型に合わせて、エラーはこのフォーマットで返して」まで具体化すると一発で通った。

具体的には:

1. **ファイルパスと型を明示する** — 「Supabaseのサーバークライアントを使って」ではなく「`src/lib/supabase/server.ts` の `createClient` を使って」
2. **既存コードのパターンを参照させる** — 「`src/app/api/terms/route.ts` と同じパターンで」
3. **判断基準を明示する** — 「コスト最小化優先」「UXはDuolingo参考」

## 一番苦労したこと — 収益化設計と集客の導線

技術的に詰まった部分はClaude Codeが大体なんとかしてくれた。本当に苦労したのは **「どうやってお金を払ってもらうか」** と **「最初のユーザーをどう集めるか」** だ。

### 収益化 — Free と Pro の線引き

個人開発SaaSで一番難しいのは「どこから有料にするか」の線引きだと思う。

DevDexでは **Free / Pro / Enterprise** の3段階のロール設計にした。Freeでも用語管理・診断・ダッシュボードは全部使える。Proで解放されるのは「アウトプット系」— 公開プロフィールURL、PDF/Excelエクスポート、診断の詳細分析、AI面談シミュレーターなど。

考え方はシンプルで、**「データを入れるのは無料、外に出すのは有料」**。使い込んでデータが溜まるほどProの価値が上がる設計にした。用語20件の上限も同じ発想で、少量なら無料で試せるけど、本気で使うならProが必要になる。

会員種別ごとの操作制限は `FeatureGate` コンポーネントで一元管理している。Freeユーザーには「この機能はProで使えます」とロック表示+アップグレード導線を出す。**制限を"壁"ではなく"試食"にする**のがポイントだった。

### 集客 — ダーツプロ × エンジニアという掛け算

自分にはちょっと変わった強みがある。**ダーツのプロ選手**でもあるということだ。

Xのフォロワーは約1,700人。ダーツ専用アカウントではなく、日頃から技術的な発信もしている。つまりフォロワーの中にエンジニアが一定数いて、かつ「あのダーツの人が作ったサービス」というキャラクター性で記憶に残りやすい。

技術系アカウントが溢れるタイムラインで、**「ダーツプロが5日でSaaS作った」** は純粋に目を引く。この「引っかかり」を意図的に使う。

### 登録→課金の導線設計

ここが一番設計に苦労した。最終的に落ち着いた導線はこうだ:

```
エンジニアタイプ診断（登録不要・64問）
  ↓ 結果ページでSNSシェアボタン
  ↓ シェアされた診断結果を見た人が診断を受ける（ループ）
  ↓ 「詳細分析を見るには登録」で会員登録
  ↓ 用語を登録して使い込む
  ↓ 20件上限 or エクスポートしたい → Pro課金
```

**登録不要で診断できる**のが最大のポイント。フリクションゼロで価値を体験させ、SNSシェアで勝手に拡散される設計にした。診断結果のシェアカードも16Personalities風にリッチにして、「自分のタイプを見せたくなる」心理を狙っている。

バズ狙いで大事なのは、**プロダクト自体にシェアされる仕組みを埋め込む**こと。告知ツイートだけに頼らない。

## まとめ — AI駆動開発で変わったこと

一番驚いたのは **「意思決定の速さが律速になる」** こと。

コードを書く速度はAIが解決した。残ったボトルネックは「何を作るか・どこまで作るか」の判断だった。5日間で520以上のPRをマージできたのは、コードが速く書けたからではなく、**判断を速くできたから**だ。

AI駆動開発の限界は **「文脈の維持」**。長い会話になるほど前の決定を忘れる。`docs/decisions/` に設計判断を記録し続けたのはそのための対策だった。24個の設計判断書がプロジェクトの「記憶」として機能し、エージェントが過去の判断と矛盾する実装をしなくなった。

:::message
Claude Codeは「楽をするツール」ではなく **「考える速度を上げるツール」** だ。使いこなすほど、自分の判断力が問われる。
:::
