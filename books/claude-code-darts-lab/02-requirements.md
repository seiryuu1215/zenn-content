## 要件定義の出発点は「不満リスト」だった

前章で書いたように、既存のダーツアプリへの不満は山ほどあった。ただ、不満をそのまま機能に変換すると「あれもこれも」になって破綻する。個人開発で最も危険なのは、スコープが膨張して完成しないことだ。

そこで Claude Code に最初に投げたのは、こんなプロンプトだった。

```
ダーツのスタッツ管理アプリを作りたい。
以下が既存アプリへの不満。これを構造化して要件定義書にまとめてほしい。
- スタッツが見るだけで分析できない
- セッティング管理が写真とメモに分散
- バレル比較に複数ECサイトを横断する必要がある
- ショップ情報がまとまっていない
```

Claude Code は不満リストを受け取ると、まず「誰がどんな場面で困っているのか」を整理する方向に誘導してきた。最初から機能一覧を出すのではなく、ペルソナとユーザーストーリーから固めるアプローチだ。

正直、最初は自分以外のペルソナなんて想定していなかった。そもそもの動機が「自分が便利に使えるアプリを作る」＋「バックエンドの技術キャッチアップとAI駆動開発の経験を積む」だったからだ。ただ、ペルソナ定義を提案された瞬間に、複合的な視点の必要性を感じた。どうせ作るなら、自分以外のプレイヤーにも使ってもらえるものにしたい——そう思えたのは、AIに「誰のため？」を問われたからだ。

## 3つのペルソナで「誰のため」を明確にする

最終的に定義したペルソナは3人。

| ペルソナ | 属性 | 主な利用シーン |
|---------|------|--------------|
| Aさん（初心者） | ダーツ歴3ヶ月、Rt.3 | 他人のセッティングを参考にしたい、成長を記録したい |
| Bさん（中級者） | ダーツ歴2年、Rt.8 | バレル買い替え検討、スタッツ推移の分析 |
| Cさん（上級者） | ダーツ歴5年、Rt.14 | セッティング情報の発信、記事執筆 |

自分自身はBさんとCさんの中間あたりだが、ポイントは **Aさん（初心者）を含めた** ことだ。自分のためだけに作ると視野が狭くなる。「ダーツを始めたばかりの人が、他のプレイヤーのセッティングを見て参考にする」というシーンを想定することで、セッティングの公開・閲覧機能の優先度が上がった。

実際、ダーツの選び方やセッティングは奥が深く、初心者が最も詰まるポイントだ。Web上には「短いダーツのほうが直線的に飛ぶ」と書いてあるサイトがある一方で、「長いほうが直線的」と真逆のことを書いているサイトもある。物理法則に基づけば、長いダーツのほうが放物線が大きくなるというのが正しい。こうした誤情報が散在している現状を見て、正しい知識を発信できる記事機能の優先度も上げることにした。

## ユーザーストーリーから機能を導く

ペルソナごとにユーザーストーリーを洗い出し、それを機能に変換していった。Claude Code との対話で出てきたストーリーの一部がこれだ。

```
【Bさんのストーリー】
- Bさんは、今使っているバレルに近い重量・形状の別ブランドを探したい
  → バレルDB + スペック検索 + レコメンド機能
- Bさんは、先月と今月のスタッツ推移を比較して成長を確認したい
  → スタッツ期間比較 + 移動平均トレンド
- Bさんは、練習後にどのスキルが弱いか客観的に知りたい
  → スキルレーダーチャート + パフォーマンスインサイト
```

この作業を3ペルソナ分繰り返すと、機能候補が40以上出てきた。ここからが本当の勝負——MVP（Minimum Viable Product）のスコーピングだ。

## MVP スコーピングの判断基準

全部を一度に作るのは無理なので、Claude Code と一緒に優先度を3段階に分けた。

```
Priority 1（MVP）: ないとアプリとして成立しない
  - ユーザー認証（メール/Google）
  - セッティング登録・一覧・詳細
  - バレルDB・検索
  - スタッツ記録（手動入力）

Priority 2（初期リリース後）: ユーザー価値が高い
  - DARTSLIVE連携（自動スタッツ取得）
  - スキル分析（レーダーチャート）
  - ショップ管理

Priority 3（成長フェーズ）: あると差別化できる
  - Stripe決済（PROプラン）
  - LINE通知連携
  - XP/ランクシステム
  - 記事投稿機能
```

判断基準はシンプルで、「これがないと毎日使わないか？」で切った。自分自身がユーザーだから、この判断はブレなかった。

AIとはいえ、ダーツのドメイン知識はこちらのほうが上だ。そこで、自分の知識とWeb上の情報をAIにインプットさせ、比較・整理させたうえで複数のプランを提示してもらい、そこから選ぶスタイルに落ち着いた。ただ、スコープを絞らないとAIが違う方向に突き進んでしまうことも学んだ。「何を聞くか」のプロンプト設計が、要件定義フェーズでは最も重要だった。

## AIとの要件定義で得た教訓

Claude Code と要件定義書を作って気づいたのは、**AIは「構造化」が異常に得意** だということだ。散らばった不満やアイデアを、ペルソナ・ユーザーストーリー・機能一覧・優先度マトリクスという形に整理するスピードは人間の比ではない。

一方で、**ドメイン知識はこちらが持ち込む必要がある**。「CONDOR AXE はシャフトとフライトが一体型だから専用の入力フォームが要る」「DARTSLIVEのフライト階級はC〜SAまである」——こうした判断は、プレイヤーでなければ出てこない。

最終的に出来上がった要件定義書は、23の機能定義（F-001〜F-023）と4つの非機能要件を含む、400行超のドキュメントになった。このドキュメントが以降の全フェーズの土台になる。
