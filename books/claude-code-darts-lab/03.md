## 技術選定の大前提：個人開発 × サーバーレス

技術選定で最初に決めたのは「サーバーを持たない」ことだった。フリーランスの傍ら一人で運用するアプリに、VPS の監視やスケーリングの手間をかける余裕はない。サーバーレスファースト——これが全ての選定の出発点だ。

この方針から、アーキテクチャは自然と決まった。

```
クライアント (Next.js)
    ↓ HTTPS
Vercel (Serverless Functions + Edge CDN)
    ↓
Firebase (Auth / Firestore / Storage)
    ↓
外部サービス (Stripe / LINE / DARTSLIVE / Sentry)
```

## なぜ Next.js か

React ベースのフレームワークは複数あるが、Next.js を選んだ理由は明確だった。

| 判断基準 | Next.js の強み |
|---------|--------------|
| Vercel との親和性 | デプロイが `git push` だけで完結する |
| App Router | ファイルベースルーティングで40ページを直感的に管理 |
| API Routes | 別途バックエンドサーバーを立てずにサーバーサイド処理が書ける |
| エコシステム | NextAuth.js、Serwist（PWA）など周辺ツールが揃っている |

ただし、ひとつ大きな判断がある。**全ページを `'use client'` で実装した**。

Next.js の App Router は Server Components を推奨しているが、darts Lab は認証状態に依存するページがほぼ全てだ。「ログインしているか」「ロールは何か」で表示内容が変わるため、Server Components のメリットが薄い。むしろ、Server Components と Client Components の境界管理でハマるリスクを避けた。

過去に携わった電子新聞プロジェクトでは、SSRで静的部分を描画し、動的なパーツだけを後から表示するアプローチを取っていた。darts Lab でも同じ方針を検討したが、ほぼ全ページがユーザー権限による表示制御を持つため、Server Components で描画してもクライアント側で再制御が必要になる。かつ、クライアント側の処理自体はそこまで複雑ではない。であれば最初から `'use client'` 主体で統一したほうがシンプルだ、という判断になった。

## なぜ Firestore か（PostgreSQL ではなく）

データベースの選定は一番迷ったポイントだ。Supabase（PostgreSQL）も有力候補だった。

| 比較軸 | Firestore | PostgreSQL (Supabase) |
|-------|-----------|----------------------|
| スキーマ | スキーマレス（変更が容易） | マイグレーション管理が必要 |
| JOIN | 非対応（非正規化で対応） | 強力なJOIN |
| リアルタイム | ネイティブ対応 | Realtime拡張が必要 |
| セキュリティ | セキュリティルール | RLS（Row Level Security） |
| 料金 | 読み書き回数課金 | 接続時間課金 |
| 認証統合 | Firebase Auth と一体 | 別途認証が必要 |

Firestore を選んだ最大の理由は **Firebase Auth との一体性** だ。認証とデータベースが同一プラットフォームで、セキュリティルールで `request.auth.uid == resource.data.userId` と書くだけでアクセス制御ができる。個人開発でこのシンプルさは大きい。

トレードオフとして、JOIN がないため非正規化が必要になる。ユーザー名やアバターURLを各ドキュメントに埋め込む設計だ。更新時の整合性は犠牲になるが、読み取りパフォーマンスは圧倒的に有利。

## デュアル認証という設計判断

認証は Firebase Auth 単独ではなく、**NextAuth.js と Firebase Auth のデュアル構成** にした。

```
ユーザー → NextAuth.js → Firebase Auth
                ↓
          JWT (role入り)
                ↓
      クライアント: NextAuth Session
      サーバー: Firebase Admin SDK
```

なぜわざわざ2つ使うのか。Firebase Auth だけでは「ロール（admin/pro/general）」をトークンに含められない。NextAuth.js を挟むことで、JWT のペイロードに `role` を載せ、クライアントサイドでもサーバーサイドでも一貫した認可判定ができる。

実はこのデュアル認証構成は、アーキテクチャ選定のタイミングで Claude Code 側から提案があったものだ。Firebase は認証だけでなく Firestore・Storage・セキュリティルールなど周辺機能が充実しており、シンプルな構成に収まる。加えて無料枠が十分にある。これらの利点を Claude Code が整理して提示してくれたことで、迷いなく採用できた。

## 状態管理：なぜ Redux も Zustand も使わないか

darts Lab には Redux も Zustand も入っていない。理由は単純で、**グローバル状態がほぼ不要** だったからだ。

- 認証状態 → NextAuth.js の `useSession`
- データ取得 → 各コンポーネントで Firestore SDK を直接呼ぶ
- UIの一時状態 → React の `useState`
- 表示設定の永続化 → `localStorage`

状態管理ライブラリを入れると、それ自体が学習コストと保守コストになる。106コンポーネントの規模でも、上記の組み合わせで十分に回っている。

## コストの現実

個人開発で避けて通れないのがコストだ。darts Lab の月額コストはこうなっている。

| サービス | 料金 | 備考 |
|---------|------|------|
| Vercel | 無料枠 | Hobby プラン |
| Firebase | 無料枠 | Spark プラン（Blaze への変更は将来） |
| Upstash Redis | 無料枠 | レートリミット用 |
| Sentry | 無料枠 | エラー監視 |
| ドメイン | 約1,500円/年 | 唯一の固定費 |

**月額ほぼ0円**。これがサーバーレスアーキテクチャの威力だ。VPS を借りていたら月数千円は確実にかかる。個人開発はランニングコストが低いほど続けられる。

Firebase を選んだ理由の一つがまさにこの無料枠だ。認証も DB もホスティングもまとめて無料枠で収まる構成を、技術選定の段階で Claude Code が提案してくれた。個人開発は「続けられるかどうか」が最大のハードルであり、ランニングコストが限りなくゼロに近いことは継続の前提条件だと考えている。
