## なぜ個人開発で10本もドキュメントを書いたのか

「個人開発なのにドキュメント10本？　オーバーエンジニアリングでは？」と思うかもしれない。自分も最初はそう思っていた。

しかし、AI駆動開発ではドキュメントの位置づけが根本的に変わる。ドキュメントは **人間が読むためだけでなく、AIがコンテキストを理解するための入力** になる。Claude Code に「セッティング登録画面を実装して」と頼むとき、要件定義書・基本設計書・セキュリティレビューが手元にあるかないかで、出力の品質がまるで違う。

## 10本のドキュメント全体像

最終的に整備したドキュメントは以下の10本だ。

```
docs/
├── 01-requirements.md        # 要件定義書（400行超）
├── 02-basic-design.md         # 基本設計書（760行超）
├── 03-detailed-design.md      # 詳細設計書
├── 04-task-breakdown.md       # タスク分解
├── 05-security-review.md      # セキュリティレビュー
├── DEVELOPER-REFERENCE.md     # 開発者リファレンス
├── ROLES-AND-PLANS.md         # ロール・権限・プラン設計
├── CRON.md                    # 自動処理（Cronジョブ）
├── ARCHITECTURE.md            # アーキテクチャ（→02に統合済み）
└── test-accounts.md           # テストアカウント情報
```

それぞれが明確な役割を持っている。

## 各ドキュメントの役割

### 01-requirements.md — 何を作るか

要件定義書。前章で書いた通り、3ペルソナ・23機能定義（F-001〜F-023）・非機能要件を網羅する。全てのドキュメントの出発点であり、「この機能は要件定義のどこに根拠があるか」を常に辿れる状態にしてある。

### 02-basic-design.md — どう作るか

基本設計書。技術選定・システム構成図・Firestoreコレクション設計・認証フロー・外部連携設計・ディレクトリ構成を含む、最も参照頻度が高いドキュメントだ。Claude Code に新機能の実装を頼むとき、このファイルをコンテキストに含めるだけで「Firestoreのどのコレクションに書くか」「認証はどのフローか」を正しく判断してくれる。

### 05-security-review.md — 安全に作れているか

17観点のセキュリティレビュー。総合評価は **A-**。認証・認可・XSS・CSRF・SSRF・入力バリデーション・Webhook署名検証・レートリミットなどを体系的にチェックしている。個人開発だからこそ、セキュリティは属人化しやすい。ドキュメントにしておくことで、抜け漏れを防いでいる。

### ROLES-AND-PLANS.md — 誰が何をできるか

ロール定義と機能別アクセス権限マトリクスのSSoT（Single Source of Truth）。`admin` / `pro` / `general` の3ロールで何ができるかを一覧化している。実装コード（`lib/permissions.ts`）とこのドキュメントが常に同期しているのがポイントだ。

### CRON.md — 自動で何が動くか

日次バッチ処理の仕様書。DARTSLIVEスタッツの自動取得、XP/実績の付与、目標の自動達成判定、LINE通知——これらが毎朝10時（JST）に自動実行される。処理フローが9ステップあり、どの順番で何が起きるかをこのドキュメントで把握できる。

### DEVELOPER-REFERENCE.md — コードの歩き方

新しい開発者（または未来の自分、またはAI）がコードベースを理解するためのガイド。「まず `types/index.ts` を読め、次に `lib/permissions.ts`」という読む順序の推奨から始まり、認証・データ層・主要コンポーネントの実装パターンまでをカバーしている。

## SSOT（Single Source of Truth）の設計

10本のドキュメントで最も意識したのは **情報の重複を避ける** ことだ。

```
要件の正規ソース      → 01-requirements.md
設計の正規ソース      → 02-basic-design.md
権限の正規ソース      → ROLES-AND-PLANS.md
セキュリティの正規ソース → 05-security-review.md
```

たとえば ARCHITECTURE.md は基本設計書と内容が重複していたため、最終的に `02-basic-design.md` に統合して非推奨化した。ドキュメントが増えること自体は悪ではないが、同じ情報が2箇所にあると必ず乖離する。

## AIとのドキュメント作成ワークフロー

ドキュメントの作成プロセスは、概ねこうだった。

1. **自分が骨子を口頭レベルで伝える** — 「認証はFirebase Auth + NextAuthのデュアル構成で」
2. **Claude Code が構造化して下書きを生成** — 表形式、フロー図、コード例を含むドキュメント
3. **自分がレビューして修正を指示** — 「ここのセキュリティルールの例は実際のコードと違う」
4. **Claude Code が修正して最終版を出力**

このサイクルを1ドキュメントあたり2〜3往復で回した。人間だけで書くと半日かかるドキュメントが、1〜2時間で仕上がる。しかも表形式やフロー図のフォーマットが統一されるので、読みやすさも担保される。

## ドキュメントがAI開発の「効き」を決める

10本のドキュメントを整備した結果、以降の開発フェーズでは Claude Code への指示が格段にシンプルになった。

例えば「ショップ管理機能を実装して」と一言伝えるだけで、Claude Code は以下を自律的に判断する。

- **要件** → 01-requirements.md の F-012 を参照
- **データ構造** → 02-basic-design.md の shopBookmarks コレクションを参照
- **権限** → ROLES-AND-PLANS.md でログインユーザーのみと確認
- **セキュリティ** → レートリミット・入力バリデーションの基準を参照

ドキュメントは「AIへの事前インストラクション」だ。書けば書くほど、AIの出力精度が上がり、手戻りが減る。55,000行のアプリを3ヶ月で作れた最大の理由は、コードを書くスピードではなく、ドキュメントによってAIの判断精度を底上げしたことにある。
