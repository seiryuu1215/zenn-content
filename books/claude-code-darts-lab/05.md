## なぜ「二重認証」が必要だったのか

darts Lab の認証基盤は、NextAuth.js と Firebase Auth を **両方使う** 構成になっている。初めて設計を見た人には「なぜ2つも？」と思われるかもしれない。答えはシンプルで、それぞれの得意領域が違うからだ。

| 認証基盤 | 役割 |
|----------|------|
| **NextAuth.js** | ページのセッション管理。JWT を Cookie に乗せて API Route で `getServerSession()` する |
| **Firebase Auth** | Firestore セキュリティルールの `request.auth` で使う。クライアント SDK の認証状態 |

NextAuth だけだと Firestore のセキュリティルールが使えない。Firebase Auth だけだと Next.js の API Route でセッション管理が煩雑になる。両方の長所を活かすために、ログイン時に Firebase Auth で認証し、その結果を NextAuth の JWT に乗せるという設計にした。

```typescript
// lib/auth.ts — authorize() の核心部分
async authorize(credentials) {
  // 1. Firebase Auth で認証
  const userCredential = await signInWithEmailAndPassword(
    auth, credentials.email, credentials.password
  );
  // 2. Firestore から role を取得
  const userDoc = await adminDb.doc(`users/${user.uid}`).get();
  const role = userDoc.data()?.role || 'general';
  // 3. NextAuth の JWT に埋め込む
  return { id: user.uid, email: user.email, name: user.displayName, role };
}
```

## 3段階ロールシステム

ユーザーロールは `general` / `pro` / `admin` の3段階。権限判定は `lib/permissions.ts` に全て集約している。

```typescript
// lib/permissions.ts — 権限判定の一元管理
export function isPro(role: UserRole | undefined): boolean {
  return role === 'pro' || role === 'admin';
}
export function canUseDartslive(role: UserRole | undefined): boolean {
  return role === 'pro' || role === 'admin';
}
export function getSettingsLimit(role: UserRole | undefined): number | null {
  if (role === 'pro' || role === 'admin') return null; // 無制限
  return SETTINGS_LIMIT_GENERAL; // 一般: 1件
}
```

ポイントは **admin は常に pro の上位互換** という設計。`isPro()` は `admin` でも `true` を返す。これにより、権限チェックを書くたびに `role === 'pro' || role === 'admin'` と書く必要がなくなる。

## Stripe Webhook のフロー

PRO への昇格は Stripe サブスクリプションの Webhook で自動処理される。手動でのロール変更は管理画面からも可能だが、通常フローはこうだ。

```
ユーザー → /pricing → Stripe Checkout → 決済完了
    ↓
Stripe → POST /api/stripe/webhook
    ↓
checkout.session.completed イベント受信
    ↓
Firestore users/{uid}.role = 'pro' に更新
    ↓
次回 JWT リフレッシュ時に session.user.role が 'pro' に
```

Webhook ハンドラでは4種類のイベントを処理している。

| イベント | 処理 |
|----------|------|
| `checkout.session.completed` | `role: 'pro'` に昇格 + サブスク情報保存 |
| `customer.subscription.updated` | ステータス同期。`active`/`trialing` なら pro 維持 |
| `customer.subscription.deleted` | `role: 'general'` にダウングレード |
| `invoice.payment_failed` | ステータスを `past_due` に更新 |

## 冪等性の担保 — stripeEvents コレクション

Webhook は **同じイベントが複数回届く可能性がある**。Stripe の公式ドキュメントにも明記されている仕様だ。これを処理するために `stripeEvents` コレクションを用意した。

```typescript
// イベントID重複チェック（リプレイ攻撃対策 + 冪等性）
const eventRef = adminDb.doc(`stripeEvents/${event.id}`);
const eventDoc = await eventRef.get();
if (eventDoc.exists) {
  return NextResponse.json({ received: true }); // 処理済み → 200で素通り
}
// ... イベント処理後に記録
await eventRef.set({ type: event.type, processedAt: Timestamp.now() });
```

これにより、ネットワーク不安定で Stripe がリトライしても、同じユーザーが二重に PRO 化されたり、ダウングレードが二重実行されたりすることがない。

## API ミドルウェアの合成

認証チェックは `lib/api-middleware.ts` に `withAuth` / `withAdmin` / `withPermission` の3つのラッパー関数として実装した。

```typescript
// 認証だけ必要な API
export const POST = withErrorHandler(
  withAuth(async (req, ctx) => { /* ctx.userId, ctx.role が使える */ }),
  'api-name'
);

// admin 限定 API（Firestore で二重確認）
export const POST = withErrorHandler(
  withAdmin(async (req, ctx) => { /* admin 確認済み */ }),
  'admin-api'
);

// 特定権限の API（関数で柔軟に判定）
export const POST = withErrorHandler(
  withPermission(canUseDartslive, 'PRO限定機能です', handler),
  'dartslive-api'
);
```

`withAdmin` は JWT の role だけでなく、**Firestore から直接 role を再取得して二重確認** している。JWT は改ざんリスクがゼロではないため、管理者操作については DB を正とする設計だ。

## JWT リフレッシュでロール変更を反映

Stripe Webhook で Firestore の role が変わっても、ユーザーのブラウザに即座に反映されるわけではない。JWT は Cookie に保存されていて、次のリフレッシュタイミングまで古い role が残る。

これを解決するため、NextAuth の `jwt` コールバックで **毎回 Firestore から最新の role を取得** するようにした。

```typescript
callbacks: {
  async jwt({ token, user }) {
    if (user) {
      token.role = user.role; // 初回ログイン時
    } else if (token.sub) {
      // セッション更新時に Firestore から最新を取得
      const userDoc = await adminDb.doc(`users/${token.sub}`).get();
      if (userDoc.exists) {
        token.role = userDoc.data()?.role || 'general';
      }
    }
    return token;
  },
}
```

Firestore の読み取りコストは増えるが、「課金したのに PRO 機能が使えない」というUX上の致命的な問題を防ぐためのトレードオフだ。
