## 1日のワークフロー

設計ドキュメントが揃った状態で実装フェーズに入ると、Claude Code との作業は一定のリズムに落ち着いた。

実は、最初の2ヶ月はフリーランス案件と並行しており、その期間は仕様固めに専念していた。実際のコーディングに本格着手したのは案件が落ち着いてからで、**実装期間は1ヶ月弱**だ。55,000行を1ヶ月弱——これが AI 駆動開発のリアルな速度感だ。

最初は月20ドルプランで始めたが、すぐにトークンが足りなくなり Max プランに切り替えた。ピーク時には **5時間分のトークンを2回連続で使い切る** ペースで開発する日もあった。朝から Claude Code に実装を依頼し、生成されたコードをレビューし、修正を指示し、また次の機能を依頼する。このサイクルを1日中回し続けると、5時間分のトークンでは足りない。

大まかなパターンはこうだ。

1. **朝:** 今日やるタスクを決める（GitHub Issues やメモから）
2. **午前〜午後:** Claude Code に設計ドキュメントを読ませて実装を依頼
3. **夕方:** 生成されたコードをレビューし、修正・テスト
4. **夜:** ダーツの練習で実際にアプリを使い、改善点をメモ

実際に運用テストで発見した改善が LINE 連携だ。当初、LINE 通知は翌日の10時に配信する設計だったが、練習後すぐにメモしたい場面があることに気づいた。通知が届くのは翌朝——これでは遅い。そこで LINE Bot から即時スタッツ取得とメモを実行できる機能を追加した。こういった改善は、自分で投げて、自分で使って初めて見えてくる。

## AI が得意なこと

3ヶ月の開発を通して、Claude Code が圧倒的に強い領域がわかってきた。

### ボイラープレートの一撃生成

API Route の認証チェック、Firestore の CRUD、MUI コンポーネントのスケルトン。こういった「パターンが決まっている」コードは、Claude Code に任せるとほぼ一発で動くものが出てくる。

例えば「目標管理の API を作って」と依頼すると、`withAuth` ラッパーで認証チェック、Firestore の読み書き、zod バリデーション、エラーハンドリングまで含めた完成度の高いコードが返ってくる。人間が手で書くと30分かかる作業が3分で終わる。

特に印象的だったのは、リポジトリ作成から初期画面の構築までのスピードだ。プロジェクトの雛形生成、ディレクトリ構成、初期コンポーネントの配置——普通なら半日かかる立ち上げ作業が、Claude Code との対話だけであっという間に動くものが出来上がった。

### パターンの横展開

`lib/permissions.ts` に1つ権限判定関数を書くと、「同じパターンで `canExportCsv` も `canUsePushNotifications` も作って」と言えば、一貫性のあるコードが量産される。53個あるスタッツコンポーネントも、最初の数個を人間が設計すれば、残りは AI が同じ構造で展開してくれる。

### テストの生成

Vitest のテストコードは、実装コードを読ませて「このファイルのテストを書いて」と言えば、正常系・異常系・エッジケースを網羅したテストが出てくる。テストの「考え漏れ」を AI が補完してくれるのは非常に助かった。

## AI が苦手なこと

一方で、AI に丸投げするとうまくいかない領域も明確にある。

### ビジネスロジックの判断

「一般ユーザーのセッティング登録上限を何件にするか」「DARTSLIVE のデータ取得頻度をどうするか」——こういった **ドメイン知識に基づく判断** は AI にはできない。

セッティング上限を「3件」にした（後に「1件」に変更した）のは、プレイヤーとしての経験から「初心者は1セッティングで十分、PRO は試行錯誤で無制限が必要」と判断したからだ。AI に聞いても「5件くらいが一般的でしょうか」という曖昧な回答しか返ってこない。

スタッツ分析については特にこだわりが強かった。基本的な知識はブラウザで検索させて理解させていたが、分析機能の設計では詳細な情報を自分から伝えた。既存アプリで不足しているポイントと、取得可能なデータで差別化できる実装手段を、対話形式で詰めていった。結果として、納得できるスタッツ分析機能に仕上がった。

### UX の微調整

スタッツダッシュボードの「どの情報を最初に見せるか」「セクションの折りたたみ初期状態」「前回比の矢印の色使い」——こういったUXの判断は、実際にダーツを投げた後にアプリを開いて「ここが見づらい」と感じる自分の体験からしか出てこない。

具体的な例を挙げると、ダークモード時のちらつき、グラフにフォーカスした際に文字色が黒のまま読めなくなる不具合があった。さらに、iPhone 実機に Xcode からデプロイしてテストしたとき、右上のメニューを展開するとデバイスのバージョンを考慮できておらず、かなり上寄りに表示されてしまう問題も見つかった。こういった問題は、実機で触らないと絶対に気づけない。

### 複数ファイルにまたがるリファクタリング

認証フローのように NextAuth 設定、Firebase Admin SDK、Firestore セキュリティルール、API ミドルウェアが絡み合う変更は、AI に一度に任せると抜け漏れが出やすい。ファイル間の依存関係を人間が把握して、変更の順序と影響範囲を指示する必要がある。

## 設計ドキュメントが AI の性能を決める

3ヶ月で学んだ最大の教訓は、**AI の出力品質は入力の品質で決まる** ということだ。

darts Lab では10本の設計ドキュメントを事前に整備した（第4章参照）。特に効果が大きかったのは以下の3つだ。

| ドキュメント | AI への効果 |
|-------------|------------|
| 基本設計書（コレクション定義） | 「users サブコレクションに goals を追加して」→ 既存構造と一貫した設計が出る |
| ROLES-AND-PLANS.md | 「この API に権限チェックを追加して」→ `withPermission(canUseDartslive, ...)` が正しく出る |
| 詳細設計書（API 一覧） | 「新しい API を追加して」→ 既存の命名規則・エラーハンドリングパターンに合わせてくれる |

逆に、設計ドキュメントなしで「いい感じに作って」と言うと、既存のコードベースと一貫性のない実装が返ってきて、手戻りが発生する。

一度設計書を整備してからは、明らかに AI の出力精度が上がった。さらに感動したのは、コードを変更した際に関連する設計書も自動で修正してくれることだ。人間がドキュメントの更新を忘れがちな部分を AI が補ってくれる——これは予想していなかったメリットだった。

## 人間が「何を作るか」、AI が「どう作るか」

最終的に落ち着いた分担はこうだった。

```
人間（自分）の担当:
  ├── 要件定義: 何を作るか、なぜ作るか
  ├── 設計判断: コレクション構造、ロール設計、課金モデル
  ├── UX判断:  情報の優先順位、画面遷移、操作感
  ├── ドメイン知識: ダーツ特有のロジック、業界慣習
  └── コードレビュー: 生成コードの品質チェック

Claude Code の担当:
  ├── 実装: 設計に基づくコード生成
  ├── パターン展開: 1つの実装を元に類似コードを量産
  ├── テスト: テストコードの生成
  ├── リファクタリング: 指示に基づく改善
  └── ドキュメント: 設計書の構造化・整理
```

この分担に至った結論として、意思決定と仕様決め以外はほぼ AI に任せられると感じた。ただし、品質担保は人間がする必要がある。実務で AI 導入が進んでいないプロジェクトが多いのは、この「品質は誰が保証するのか」という問題が解決されていないからではないかと考えている。

この分担が機能した結果、55,000行のコードベースを3ヶ月で構築できた。もちろん AI が書いたコードの品質を人間がレビューする工程は省けない。だが「ゼロからコードを書く時間」が大幅に削減されたことで、設計やUXの判断に時間を使えるようになった。

AI 駆動開発を経験して、より上流にシフトしなければならないと強く感じた。コードしか書けないエンジニアの仕事は、近い将来不要になる——これは危機感ではなく確信だ。フリーランスとして現場で信頼を勝ち取り、意思決定や設計に近いポジションを経験すること。そして AI を活用した技術のキャッチアップを並行して続けること。この2つが、これからのエンジニアに必要な生存戦略だと考えている。
