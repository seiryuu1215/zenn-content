## セキュリティは後回しにしがち問題

個人開発でセキュリティレビューをちゃんとやっている人は、正直そんなに多くないと思う。「動けばいい」「ユーザー少ないし」——自分もそう思っていた時期がある。

## CRITICAL: 管理者メールのハードコードとSSRF

最も深刻だったのは2件。

**1つ目** は `scripts/` 配下の5ファイルに管理者メールアドレスがハードコードされていたこと。GitHub公開した瞬間にメールアドレスが世界に公開されるところだった。対策は単純で、すべて `process.env.ADMIN_EMAIL` に統一した。

**2つ目** は画像プロキシAPI（`/api/proxy-image`）のSSRF脆弱性。外部画像をプロキシする機能で、任意のURLを渡せる状態だった。これを悪用されると、サーバーから内部ネットワークにリクエストを飛ばされる可能性がある。

対策として、ドメインホワイトリストを実装した。

```typescript
// app/api/proxy-image/route.ts
const ALLOWED_DOMAINS = [
  'dartshive.jp',
  'www.dartshive.jp',
  'image.dartshive.jp',
  'firebasestorage.googleapis.com',
];

// HTTPS強制 + ドメイン検証 + Content-Type検証 + SVGブロック
if (parsed.protocol !== 'https:') return error(400);
if (!ALLOWED_DOMAINS.some(d => parsed.hostname === d)) return error(403);
```

4ドメインだけ許可、HTTP拒否、SVGブロック（XSS防止）、10MBサイズ制限。多層防御の考え方だ。

## HIGH: 暗号化とWebhook署名

DARTSLIVEの認証情報（メール/パスワード）を日次Cronで使うために保存する必要があった。平文保存は論外なので、AES-256-GCM で暗号化した。

```typescript
// lib/crypto.ts
// IV: 12バイト（毎回ランダム生成）
// 認証タグ: 16バイト（改ざん検出）
// 出力: iv + authTag + ciphertext（Hex文字列）
```

ポイントは3つ。ランダムIVで同じパスワードでも異なる暗号文になること。認証タグで改ざんを検出できること。鍵は環境変数管理でソースコードに含まれないこと。

Stripe と LINE の Webhook 署名検証も HIGH で指摘された。Stripe は `constructEvent()` で署名検証 + `stripeEvents/{eventId}` で冪等性を担保。LINE は HMAC-SHA256 + `crypto.timingSafeEqual()` でタイミング攻撃を防止している。

## MEDIUM: CSP、CSV、レートリミット

**CSP（Content Security Policy）** は nonce 方式で実装した。リクエストごとに `crypto.randomUUID()` でnonce を生成し、インラインスクリプトに付与する。

**CSVインジェクション** は、スタッツをCSVエクスポートする機能で発見された。メモ欄に `=SUM(...)` のような数式を書かれると、Excelで開いたときに実行される可能性がある。

```typescript
// 数式プレフィックスをシングルクォートで無害化
let memo = (d.memo ?? '').replace(/"/g, '""');
if (/^[=+\-@\t\r]/.test(memo)) memo = `'${memo}`;
```

**レートリミット** は当初インメモリ Map で実装していたが、Vercel のサーバーレス環境ではインスタンス間で共有されない問題があった。最終的に **Upstash Redis** による分散レートリミットに移行し、60リクエスト/分/IP の制限をグローバルに適用した。

## Firestoreセキュリティルール — 最後の砦

約200行の `firestore.rules` は、クライアントから直接 Firestore にアクセスするこのアプリにおいて最後の砦だ。

特に重要なのが **ロール昇格防止** のルール。

```javascript
allow update: if isOwner(userId)
  && !request.resource.data.diff(resource.data).affectedKeys()
      .hasAny(['role', 'stripeCustomerId', 'subscriptionId']);
```

ユーザーが自分のプロフィールを更新するとき、`role` や `stripeCustomerId` を書き換えられないようにしている。これがなければ、ブラウザの開発者ツールから `role: 'admin'` に書き換えるだけで管理者権限を奪取できてしまう。

## 総合評価: A-

全11件の指摘を対応した結果、総合評価は **A-** となった。重大脆弱性なし、全中優先度項目対応済み。

個人開発でもセキュリティレビューは必須だ。そしてAIはこの領域で非常に頼りになる。チェックリストを体系的に作り、見落としがちな観点を網羅的に洗い出してくれる。ただし、最終的に「この対策で十分か」を判断するのは人間の責任だ。AIの指摘を鵜呑みにするのではなく、自分のアプリの規模やリスクに合わせて優先度をつけることが大事だと感じた。
