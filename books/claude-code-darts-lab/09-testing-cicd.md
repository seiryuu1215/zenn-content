## 「全部テストする」は幻想だった

82,000行のコードベースに対して、テスト数は636（ユニットテスト）+ 240（Storybookストーリー）。初期段階では164テストから始まり、機能追加のたびにテストを積み上げてきた。

個人開発では「全部テストする」時間はない。だからこそ、何を守るかの選別が重要になる。

結論から言うと、「壊れたら致命的な場所」だけを手厚くテストし、それ以外は割り切った。限られた時間のなかで品質を担保するには、テスト対象の選別が最も重要な設計判断だった。

## テストするもの・しないもの

### テストしたもの（636+ ユニットテスト / Vitest）

| ファイル | 内容 | テスト数 |
|---------|------|---------|
| `api-middleware.test.ts` | 認証・権限・エラーハンドリング | 10 |
| `permissions.test.ts` | 全権限関数の正常系/異常系 | ~20 |
| `dartslive-percentile.test.ts` | パーセンタイル計算・補間 | ~15 |
| `webhook/route.test.ts` | Stripe署名検証・イベント処理 | 6 |
| `goals.test.ts` | 目標定義・進捗計算ヘルパー | ~30 |
| `xp-engine.test.ts` | レベル計算・ランク判定 | ~20 |
| `xp-engine.test.ts`（progression） | Cron XP差分計算・実績チェック | ~25 |

選定基準は明確だ。**「バグがあったらユーザーのデータや課金に影響するロジック」** を優先した。

- **権限管理（permissions）**: `isPro()` のバグでフリーユーザーが有料機能を使えたら致命的
- **Webhook処理**: Stripe の Webhook を間違えたら課金状態がズレる
- **XPエンジン**: 計算ミスでレベルが狂ったらユーザー体験が崩壊する
- **目標システム**: 達成判定のバグで「達成したのにXPがもらえない」は最悪

### テストしなかったもの（意図的なトレードオフ）

- **Reactコンポーネント**: React Testing Library を導入しなかった
- **Firestoreセキュリティルール**: Firebase Emulator を使ったテストは未実施
- **E2Eテスト**: Playwright / Cypress は導入しなかった
- **スクレイピング処理**: DARTSLIVE の自動データ取得のモックテストなし

Reactコンポーネントをテストしなかった理由は、コストパフォーマンスだ。UIの見た目は Storybook で確認でき、ロジックはカスタムフックや `lib/` に切り出してある。コンポーネントのテストを書いても、MUIのレンダリング結果に依存する脆いテストになりがちで、メンテナンスコストに見合わないと判断した。

E2Eテストも同様だ。一人開発では「ブラウザを開いて手動で確認する」ほうが、E2Eテストの環境構築とメンテナンスより速い場面が多い。

## Storybook — 240ストーリーのUIカタログ

Reactコンポーネントのテストを書かない代わりに、Storybook を導入した。

Storybook は実務で使った経験があり、操作性も個人的に好きだった。今回のアプリでは幅広いスタッツのユーザーが想定され、データの値によってUIがかなり変わる。複数パターンのUIチェックをモックデータで手軽にできる点が採用の決め手だ。

16のストーリーを作成し、主にスタッツ画面のチャートコンポーネントやカード類をカタログ化した。新しいデザインを試すとき、本番データに依存せずにモックデータで表示確認できるのが便利だった。

## CI/CD: 4段階ゲートの設計

GitHub Actions で **4段階のゲート** を設けた。

```
lint → format → test → build
```

この順序には意味がある。

1. **lint（ESLint）**: 構文エラーや未使用変数を最速でキャッチ
2. **format（Prettier）**: コードスタイルの統一チェック
3. **test（Vitest）**: ユニットテスト実行（636+テスト）
4. **build（next build）**: TypeScript型チェック + ビルド成功確認

lint で落ちればそれ以降は実行しない。最も軽いチェックから順に実行することで、フィードバックを早く返す。

Vercel との連携はシンプルだ。GitHub にプッシュすると Vercel が自動でプレビューデプロイを作成し、CI が通ったら本番にデプロイする。この流れに特別な設定はほとんど不要だった。

## Dependabot との付き合い方

Dependabot を有効にしたら、毎週15件前後のPRが飛んでくるようになった。

実務での運用フローを参考にしている。patch（軽微な修正）は隙間時間に自分でレビューしてマージ。minor は内容を確認して重み付けし、チームのタスク状況に応じて割り振る。major はリーダークラスに依頼するか、技術力の高いメンバーに任せることが多かった。緊急のセキュリティアップデートが入った際には自分がすぐ対応する——この優先度判断は日々のチェックの中で身につけたものだ。

依存ライブラリの更新は地味だが重要だ。`npm audit` で脆弱性0件を維持するのはセキュリティの基本であり、これは前章で触れたセキュリティレビューの「依存ライブラリ」観点と直結している。

## AIとテスト — 何が得意で何が苦手か

テスト生成は Claude Code が最も得意とする作業の一つだった。

「`permissions.ts` の全関数に対してユニットテストを書いて」と指示すると、正常系・異常系・エッジケースを網羅したテストが一発で生成される。テストの「パターン」は決まっているので、AIが得意とする定型作業にぴったりだ。

一方で、**何をテストすべきか** の判断はAIには難しい。「このアプリで壊れたら一番困る場所はどこか」は、ドメイン知識とビジネス判断の領域だ。テスト戦略の設計は人間がやり、テストコードの実装はAIに任せる——この分担が最も効率的だった。

```typescript
// AIが生成したテストの例（xp-engine.test.ts）
describe('calculateLevel', () => {
  it('XP 0 はレベル1', () => {
    expect(calculateLevel(0)).toBe(1);
  });
  it('XP 100 はレベル2', () => {
    expect(calculateLevel(100)).toBe(2);
  });
  it('境界値: レベルアップ直前', () => {
    expect(calculateLevel(99)).toBe(1);
  });
  // ... AIが境界値を自動的に洗い出してくれる
});
```

## 636テストに至った経緯

最初は164テストからスタートした。「壊れたら致命的な場所」だけを手厚くカバーする方針だ。権限チェック、課金処理、XP計算——これらはユニットテストで確実に防げていた。

しかし機能が増えるにつれ、テストの重要性を痛感した。全てのロジックを目視レビューするのは追いつかない。だからこそ機能追加のたびにテストケースを積み上げ、現在は636のユニットテスト + 240のStorybookストーリーで品質を担保している。

UIの表示崩れやレスポンシブの問題は Storybook の全ページ・全コンポーネントカバレッジで対応。一人開発でもここまでテストを厚くできたのは、AIがテストコード生成を得意としているからだ。
