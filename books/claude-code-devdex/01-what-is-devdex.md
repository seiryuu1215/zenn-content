## スキルシートがエンジニアの外面なら、DevDexは内面

エンジニアなら誰でも「スキルシート」を書いたことがあるだろう。使用技術、経験年数、プロジェクト概要。フォーマットに沿って埋めていく、いわばエンジニアの"外面"だ。

でも、面談の直前にスキルシートを見返して「これ、ちゃんと説明できるかな」と不安になったことはないだろうか。

「React 3年」と書いたけど、Server Componentsの仕組みをちゃんと説明できるか。「Docker経験あり」と書いたけど、マルチステージビルドのメリットを聞かれたらどうしよう。スキルシートには書いてある。でも、自分の理解度には自信がない。

DevDexは、その「内面」を管理するツールだ。

IT用語ひとつひとつに対して、自分がどの程度理解しているかを5段階で記録する。概要はAIが補完してくれるが、「自分の経験」と「面談で話せるサマリー」は自分で書く。コピペでもいい。大事なのは、自分の言葉で理解度を言語化するプロセスだ。

このアプリを作った動機、技術スタック、そしてなぜ「サブエージェント5体制」という開発手法に至ったのか。第1章ではDevDexの全体像を描く。

## なぜDevDexを作ったのか

### ペルソナ: 25歳フリーランスエンジニア

DevDexのメインターゲットは、実務経験2〜3年の若手エンジニアだ。ペルソナとして想定したのは、25歳のフリーランスエンジニア。

このペルソナには具体的な課題がある。

- **面談前のキャッチアップ不安**: 案件の面談前に「この技術、ちゃんと話せるかな」と焦る
- **知識の散在**: Notion、Qiita、ブックマーク、脳内。学んだことがバラバラに存在する
- **成長実感の欠如**: 1年前と比べて何が成長したのか、定量的に把握できない

逆に、シニアエンジニア（経験10年超）も対象にしている。ただし彼らは「一から用語を登録していく」のが面倒なので、長文貼り付けからのAI用語一括抽出機能で対応した。職務経歴書やプロジェクト概要をコピペすれば、関連する技術用語が自動で抽出・登録される。

### 前作「ダーツラボ」からの技術的成長

DevDexは自分にとって2作目のフルスタックWebアプリだ。1作目は「ダーツラボ」。プロダーツ選手としての練習管理アプリで、以下の技術スタックで構築した。

| | ダーツラボ | DevDex |
|---|---|---|
| フレームワーク | Next.js (Pages Router) | Next.js 16 (App Router) |
| DB/Auth | Firebase (Firestore + Auth) | Supabase (PostgreSQL + Auth) |
| UIライブラリ | MUI (Material UI) | shadcn/ui + Tailwind CSS v4 |
| テスト | Jest | Vitest 4 |
| AI活用 | なし | Anthropic API (Claude) |
| 開発手法 | Claude Codeとペアプロ | Claude Code サブエージェント5体制 |

ダーツラボではFirebaseを使ったが、Firestoreのスキーマレスな性質に苦労した。リレーショナルなデータ（ユーザー、スタッツ、セッティング、フレンド）を扱うのにNoSQLは向いていないと実感した。DevDexではSupabaseのPostgreSQLを採用し、RLS（Row Level Security）で行レベルのアクセス制御も実現している。

UIライブラリもMUIからshadcn/ui + Tailwind CSSに移行した。MUIは包括的だが重い。shadcn/uiはコンポーネントのソースコードが手元にあるため、カスタマイズの自由度が段違いだ。

そして最大の変化は開発手法。ダーツラボではClaude Codeと1対1のペアプログラミングだったが、DevDexでは5つのサブエージェント（PM・実装・テスト・レビュー・日記）を設計し、役割分担で開発を進めた。この書籍のメインテーマだ。

## DevDexの機能概要

DevDexは「IT用語の理解度管理」を起点に、エンジニアのキャリア支援まで広がるアプリだ。v0からv3まで段階的に構築した。

### 用語管理（v0: MVP）

アプリの核。IT用語を登録し、以下の情報を管理する。

- **概要**: AI補完またはマスターデータから自動入力
- **自分の経験**: 「実務でReactを2年使い、SPAからSSRへの移行を経験」のように自分の言葉で記述
- **面談サマリー**: 面談で聞かれたときに話す内容を整理
- **習熟度スコア**: 1（聞いたことがある）〜 5（教えられる）の5段階
- **ピン留め/ウォッチ**: 重要な用語を固定表示、気になる用語をウォッチリストに追加

用語はカテゴリ（frontend, backend, infra, db, other）で分類し、親子関係と関連用語のリンクも管理できる。

### AI用語抽出（v0）

Anthropic API（Claude）を活用した目玉機能。職務経歴書やプロジェクト概要を丸ごと貼り付けると、含まれるIT用語を自動抽出する。シニアエンジニアが「一から登録するのが面倒」という課題を解決するための機能だ。

### エンジニアタイプ診断（v2）

4つの軸で合計16タイプのエンジニアタイプを診断する。さらにMBTIの16タイプと組み合わせた256パターンのプロファイルを用意した。

- **スコープ**: 特化型 vs 汎用型
- **アプローチ**: 理論型 vs 実践型
- **フォーカス**: 品質型 vs スピード型
- **スタイル**: チーム型 vs 個人型

診断結果は公開プロフィールに表示でき、面談時の自己紹介にも使える。

### スキルシート・案件マッチング（v2〜v3）

登録した用語データをベースに、スキルシートを自動生成する。経歴、スキルセット、自己PRをフォームで入力し、PDFやExcelでエクスポートできる。さらに、スキルシートの内容と案件の要件を照合する案件マッチング機能も実装した。

### 企業向け機能（v3）

toB向けの収益化機能。組織（企業）を作成し、メンバーを招待して候補者を検索・比較できる。組織ロール（owner, admin, member, viewer）による権限制御、候補者のブックマークやアクティビティログも備える。

## 技術スタックの選定理由

### Next.js 16 (App Router)

React Server Componentsを活用し、クライアントに送るJavaScriptを最小限にする設計。`'use client'`は本当に必要なコンポーネント（フォーム、インタラクティブUI）にだけ付与する。

App Routerのファイルシステムベースのルーティングは直感的で、`layout.tsx`によるネストレイアウトが強力。認証チェックをlayoutで一括処理できるのも大きい。

### Supabase (PostgreSQL + Auth + RLS)

Firebaseから移行した最大の理由はリレーショナルデータベース。用語、ユーザー、関連用語、組織、メンバーシップなど、リレーションが複雑なデータモデルにはPostgreSQLが適している。

RLS（Row Level Security）により、データベースレベルでアクセス制御を実現。APIで認可ロジックを書き忘れても、RLSが最後の防衛線になる。22本のマイグレーションでスキーマを段階的に進化させた。

### Tailwind CSS v4 + shadcn/ui

Tailwind CSS v4は設定ファイルが不要になり、CSSファイルに直接設定を記述できるようになった。shadcn/uiのコンポーネントはプロジェクトにソースコードとして取り込まれるため、細かいカスタマイズが容易。ダークモードもCSS変数の切り替えで実現している。

### Vitest 4

Jest互換のAPIで学習コストゼロ。ESModulesネイティブ対応で、Next.jsとの相性が良い。1,400件超のテストが高速に実行できるのはVitestのおかげだ。

### Anthropic API (Claude)

用語の概要補完と長文からの用語抽出に使用。構造化された出力（JSON）を安定して返すため、パース処理がシンプルに書ける。

## 規模感: 数字で見るDevDex

DevDexの最終的な規模を数字で示す。

| 指標 | 数値 |
|---|---|
| APIエンドポイント | 38本 |
| ページ数 | 20画面 |
| ユニットテスト | 1,400件超 |
| DBマイグレーション | 22本 |
| 用語辞書 | 117件（プリセット） |
| エンジニアタイプ | 16タイプ x 16 MBTI = 256パターン |
| 個人ロール | 3種類（user, pro, admin） |
| 組織ロール | 4種類（owner, admin, member, viewer） |

この規模のアプリを、サブエージェント5体制でどう開発したのか。次章以降で詳しく解説する。

## この書籍の読み方

この書籍は「Claude Codeをどう使いこなすか」に焦点を当てている。

特にサブエージェント（PM・実装・テスト・レビュー・日記）の設計と運用が中心テーマだ。各エージェントの役割、プロンプト設計、成果物、そして実際の運用で発生した問題と解決策を包み隠さず記録した。

**想定読者**:

- Claude Codeを使い始めたが、もっと効率的に使いたいエンジニア
- サブエージェントの概念は知っているが、実際にどう設計すればいいか分からない人
- AI駆動開発で中〜大規模なアプリを作りたい人
- Next.js + Supabaseの実践的な構成に興味がある人

**各章の概要**:

- **第2章**: サブエージェント5体制の全体設計。なぜ5つに分けたのか
- **第3章〜第6章**: 各エージェント（PM・実装・テスト・レビュー）の詳細
- **第7章**: ログから日記、日記からZenn記事への変換パイプライン
- **第8章**: プロンプト設計とCLAUDE.md。Skills、MEMORY.md、Compact Instructions
- **第9章**: DevDexのアーキテクチャ。DB設計、Feature Gate、認証・認可
- **第10章**: 振り返りと次のステップ。数値、課題、今後の展望

では、第2章でサブエージェントの設計思想から見ていこう。
