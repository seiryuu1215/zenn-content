## 7日間・78時間で96,000行のSaaSを作った

高校中退、通信制高校、専門卒。有名企業の勤務経験なし。25歳でフリーランスとして独立し、ダーツのプロ選手でもある。

そんな経歴の人間が、Claude Codeのサブエージェント6体制を使って、**7日間・78時間**で62 API・35ページ・2,350テスト・169コンポーネントのフルスタックSaaSを構築した。1日15時間超の開発を続け、Day 4-5はダーツプロイベントの合間にスマホからClaude Codeへ指示を出しながら開発を進めた。

この書籍は、その全工程の記録だ。

![DevDexの規模感](/images/devdex-scale-overview.png)

## スキルシートがエンジニアの外面なら、DevDexは内面

エンジニアなら誰でも「スキルシート」を書いたことがあるだろう。使用技術、経験年数、プロジェクト概要。フォーマットに沿って埋めていく、いわばエンジニアの"外面"だ。

でも、面談の直前にスキルシートを見返して「これ、ちゃんと説明できるかな」と不安になったことはないだろうか。

「React 3年」と書いたけど、Server Componentsの仕組みをちゃんと説明できるか。「Docker経験あり」と書いたけど、マルチステージビルドのメリットを聞かれたらどうしよう。スキルシートには書いてある。でも、自分の理解度には自信がない。

DevDexは、その「内面」を管理するツールだ。IT用語ひとつひとつに対して、自分がどの程度理解しているかを5段階で記録する。概要はAIが補完してくれるが、「自分の経験」と「面談で話せるサマリー」は自分で書く。大事なのは、自分の言葉で理解度を言語化するプロセスだ。

ターゲットは実務経験2〜3年の若手エンジニア。面談前のキャッチアップ不安、知識の散在、成長実感の欠如——この3つの課題を解決する。シニアエンジニア向けには、職務経歴書からのAI用語一括抽出で「一から登録する面倒さ」を解消した。

## 規模感: 数字で見るDevDex

個人開発のSaaSは一般的にAPI 10〜20本・画面5〜15ページ・テスト数百件程度が相場だ。DevDexの規模はその数倍に達している。

| 指標 | DevDex | 個人開発の相場 |
|---|---|---|
| APIエンドポイント | 62本 | 10〜20本 |
| ページ数 | 35画面 | 5〜15画面 |
| ユニットテスト | 2,350件 | 0〜300件 |
| DBマイグレーション | 40本 | 5〜10本 |
| Issues/PRs | 637+件 | — |

加えて、エンジニアタイプ診断は16タイプ × 16 MBTI = 256パターン、ロール設計は個人3種類 + 組織4種類。toC × toBの両面を持つSaaSとしてはフル装備に近い。

## DevDexの機能概要

この規模のアプリが何をするのか。v0からv3まで段階的に構築した機能を紹介する。

### 用語管理（v0: MVP）

アプリの核。IT用語を登録し、概要（AI補完）・自分の経験・面談サマリー・習熟度スコア（1〜5段階）を管理する。カテゴリ分類、親子関係、関連用語のリンク、ピン留め/ウォッチ機能も備える。

### AI用語抽出（v0）

職務経歴書やプロジェクト概要を丸ごと貼り付けると、含まれるIT用語をAIが自動抽出する。シニアエンジニアが「一から登録するのが面倒」という課題を解決するための機能だ。

### エンジニアタイプ診断（v2）

4つの軸（特化vs汎用・理論vs実践・品質vsスピード・チームvs個人）で16タイプを診断。MBTIとの組み合わせで256パターンのプロファイルを用意した。診断結果は公開プロフィールに表示でき、面談時の自己紹介にも使える。

![エンジニアタイプ診断結果](/images/devdex-diagnosis-result.png)
*診断結果画面。4軸スコアとタイプ情報が表示される*

### スキルシート・案件マッチング（v2〜v3）

登録した用語データからスキルシートを自動生成し、PDFやExcelでエクスポート。案件の要件との照合機能も実装した。

### 企業向け機能（v3）

toB向けの収益化機能。組織を作成し、メンバー招待・候補者検索・比較ができる。4段階の組織ロールによる権限制御を備える。

## 前作「ダーツラボ」からの進化

DevDexは2作目のフルスタックWebアプリだ。1作目の「ダーツラボ」（プロダーツ選手としての練習管理アプリ）と比較すると、技術スタックも開発手法も別物になった。

| | ダーツラボ（前作） | DevDex（今作） |
|---|---|---|
| フレームワーク | Next.js (Pages Router) | Next.js 16 (App Router) |
| DB/Auth | Firebase (Firestore + Auth) | Supabase (PostgreSQL + RLS) |
| UI | MUI | shadcn/ui + Tailwind CSS v4 |
| テスト | Jest / 300件（手動設計） | Vitest 4 / 2,350件（Agent自動生成） |
| 開発体制 | Claude Codeとペアプロ | サブエージェント6体制 |
| 意思決定記録 | なし（脳内 + チャット履歴） | 25件（docs/decisions/で構造化） |
| Issues/PRs | 約50件 | 637+件 |

最大の変化は開発手法だ。前作では「Claude Codeと一緒にコードを書く」だったのが、今作では「エージェントチームにプロジェクトを任せる」に進化した。テスト数が300→2,350に跳ね上がったのも、実装者とテスト作成者が分離されたからだ。この書籍のメインテーマはまさにここにある。

![サブエージェント6体制の構成図](/images/devdex-subagent-flow.png)
*6つのエージェントが役割分担し、人間は「方向性を決める」「判断を承認する」「最終確認する」の3つだけ*

## この書籍の読み方

この書籍は「Claude Codeをどう使いこなすか」に焦点を当てている。

特にサブエージェント（PM・実装・テスト・レビュー・日記・ビジネス）の設計と運用が中心テーマだ。各エージェントの役割、プロンプト設計、成果物、そして実際の運用で発生した問題と解決策を包み隠さず記録した。

**想定読者**:

- Claude Codeを使い始めたが、もっと効率的に使いたいエンジニア
- サブエージェントの概念は知っているが、実際にどう設計すればいいか分からない人
- AI駆動開発で中〜大規模なアプリを作りたい人
- Next.js + Supabaseの実践的な構成に興味がある人

**各章の概要**:

- **第2章**: サブエージェント6体制の全体設計。なぜ6つに分けたのか
- **第3章〜第6章**: 各エージェント（PM・実装・テスト・レビュー）の詳細
- **第7章**: ログから日記、日記からZenn記事への変換パイプライン
- **第8章**: プロンプト設計とCLAUDE.md。Skills、MEMORY.md、Compact Instructions
- **第9章**: 実装実績を数字で振り返る。Day 1〜5の開発ログ
- **第10章**: DevDexのアーキテクチャ。DB設計、Feature Gate、認証・認可
- **第11章**: 失敗と学び。AI駆動開発で踏んだ地雷10選
- **第12〜13章**: まとめと今後の展望

では、第2章でサブエージェントの設計思想から見ていこう。
