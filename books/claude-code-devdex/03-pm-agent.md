## PM Agentの役割

PM Agentは「コードを書かないエージェント」だ。

ユーザーの曖昧な要望を受け取り、実装可能な粒度のタスクに分解し、設計判断の理由を記録する。これがPM Agentの全仕事だ。コードを1行も書かない代わりに、「何を作るか」「なぜこのアプローチなのか」「何を選ばなかったのか」を徹底的にドキュメントに残す。

DevDexの開発では、PM Agentが20件の意思決定記録を出力した。v0からv3までの段階的な機能開発において、タスク分解、技術選定、権限設計、収益化設計のすべてがここに集約されている。

なぜ「コードを書かないこと」が重要なのか。実装ができるエージェントに要件整理を任せると、「技術的に楽だから」という理由で設計判断が歪む。PM Agentはコードが書けないからこそ、ユーザーの要望とプロダクトの方向性に集中できる。

## エージェント定義

PM Agentの定義は `.claude/agents/pm-agent.md` に記述する。

```markdown
---
name: pm-agent
description: ユーザーの要望を受けて要件整理・タスク分解・意思決定の記録を行う。
tools: Read, Glob
skills:
  - devdex-conventions
---

あなたはこのプロジェクトのPMです。実装はしません。

役割：
- ユーザーの要望を聞いてタスクに分解する
- 実装方針・技術選定の意思決定を記録する（採用した理由・却下した選択肢も含める）
- 作業結果を docs/decisions/YYYY-MM-DD-[topic].md に保存する
- 完了後 implement-agent に引き渡す

制約：コードを書かない / docs/decisions/ 以外のファイルを変更しない
```

注目すべきは `tools: Read, Glob` の行だ。PM Agentが使えるツールはファイルの読み取りとファイル検索だけ。Bashも、Editも、Writeも使えない。ただし `docs/decisions/` への書き込みだけは許可されている。この制限が、PM Agentを「考える専門家」に仕立てる。

`skills: devdex-conventions` によって、プロジェクト固有のコーディング規約（TypeScriptのルール、Supabaseの命名規則、コンポーネント設計方針）がPM Agentにも共有される。PM Agentがタスクを分解するとき、規約を理解した上で実装可能な指示を出せるようになる。

## 成果物: 意思決定記録

PM Agentの成果物は `docs/decisions/YYYY-MM-DD-[topic].md` 形式のMarkdownファイルだ。DevDexでは統一テンプレートを用意した。

```markdown
---
date: YYYY-MM-DD
agent: pm-agent
topic: ''
status: pending
---

## 背景・課題

## 採用した方針

## 却下した選択肢

| 選択肢 | 却下理由 |
| ------ | -------- |

## タスク分解

- [ ] タスク1
```

テンプレートは4つのセクションで構成される。

**背景・課題**: なぜこの作業が必要なのか。現状の問題は何か。

**採用した方針**: 何をどう実装するか。技術選定の理由も含める。

**却下した選択肢**: 検討したが採用しなかった代替案とその理由。

**タスク分解**: 実装タスクの一覧。工数見積もりと依存関係も記載する。

この中で最も価値が高いのは「却下した選択肢」セクションだ。1か月後に「なぜServer Actionsを使わなかったんだっけ？」と疑問に思ったとき、ここを見れば理由が書いてある。採用理由だけでなく不採用理由も記録することで、同じ議論を繰り返さずに済む。

## 実例1: v0タスク分解

DevDex v0（MVP）のタスク分解を実例として見ていく。PM Agentに渡した要望は「CLAUDE.mdのv0機能一覧を実装可能なissueに分解してほしい」という一文だけだ。

PM AgentはCLAUDE.mdを読み、以下の方針を出力した。

```markdown
## 採用した方針

### 全体方針
1. データ層 → UI層 → インタラクション層 → AI機能 → 認証 の順で積み上げる
2. 認証は最後に実装し、開発中は anon ロールで操作可能にする
3. 1 issue = 1 PR = 1〜2時間の粒度を厳守する
4. Route Handlers でCRUD APIを構築し、UIからfetchで呼び出す

### API設計方針
- Route Handlers (`src/app/api/terms/route.ts` 等) を採用
- レスポンス形式: `{ data: T | null, error: string | null }`
- Server Actions は使わない
```

そして却下した選択肢として以下を記録した。

| 選択肢 | 却下理由 |
|---|---|
| Server ActionsでCRUD | フォームと密結合しすぎる。API境界が不明確でデバッグしにくい。将来モバイル対応時に再実装が必要になる |
| 認証を最初に実装 | 全機能の開発がブロックされる。まずローカルで動くものを作り、最後に認証を被せるほうが効率的 |
| 用語登録と編集を別画面 | CLAUDE.mdの設計（詳細/編集モード切り替え）に反する。フォームコンポーネントの共有で開発効率も上がる |

Server Actionsの却下理由は、後のv3で実際にExpo（React Native）でのモバイル対応を検討した際に「あのときRoute Handlersにしておいてよかった」と実感することになる。PM Agentの時点で将来のモバイル対応を考慮した設計判断が記録されていた。

## タスク分解の粒度: 1〜2時間ルール

PM Agentに課した最も重要なルールが「1 issue = 1〜2時間で終わる粒度」だ。

v0では14個のissueに分解され、合計約18.5時間と見積もられた。

```
Phase 1: 基盤（データ層）     → 2 issue (2.5h)
Phase 2: コアUI              → 4 issue (5.5h)
Phase 3: インタラクション    → 4 issue (4h)
Phase 4: ダッシュボード      → 1 issue (1.5h)
Phase 5: AI機能              → 2 issue (3h)
Phase 6: 認証                → 1 issue (2h)
```

v1では7 issue（約9時間）、v2では10 issue（約15.5時間）と、バージョンが上がるにつれてタスクの複雑度は増したが、1 issueの粒度は常に1〜2時間に収まっている。

この粒度にこだわる理由は3つある。

**1. サブエージェントフロー1周の単位になる**。1 issue = 1 Branch = 1 PR = PM Agent → Implement Agent → Test Agent → Review Agentの1サイクル。このサイクルが小さければ、手戻りが発生しても修正範囲が限定される。

**2. コンテキストが溢れない**。Claude Codeのコンテキストウィンドウは有限だ。2時間以上かかる大きなタスクでは、実装の途中でコンテキストが圧縮され、前半の設計意図が失われることがある。1〜2時間なら1セッション内に収まる。

**3. 進捗が見える**。14個のissueがあれば、1つマージするたびに7%進んだことが分かる。1つの巨大issueで1週間作業するより、毎日2〜3個のPRがマージされていくほうが精神的にも健全だ。

## 実例2: 段階的なバージョン計画

PM Agentが出力したv0〜v2のタスク分解を並べると、段階的な成長戦略が見える。

**v0（MVP）: 14 issue / 18.5h**
データ層の構築から始め、CRUD API → UI → インタラクション → ダッシュボード → AI抽出 → 認証の順。「まず自分が使えるもの」を最短で作る。

**v1（他人に見せられるレベル）: 7 issue / 9h**
v0でMOCK_DATAを使っていた画面のAPI連携化から着手。マスターデータ一括登録、親子関係、関連用語リンクを追加。v0の土台をリアルなデータに接続する。

**v2（ゲーム・公開・共有）: 10 issue / 15.5h**
レベルシステム、公開プロフィール、PDFエクスポート、GitHub連携。ユーザーが外部に見せられる機能を追加。Phase分割で並列実装可能な設計にした。

各バージョンの却下リストも注目に値する。v2で「イラスト制作のキャラ」を却下した理由は「コスト高、外部依存」。代わりに「CSS/SVGベースの簡素なアバター（レベル帯で色変化）」を採用した。コスト0運用の方針に沿った判断がPM Agentの記録として残っている。

## 実例3: 権限設計の意思決定

DevDexの権限設計は、個人ロール3種（user, pro, admin）と組織ロール4種（owner, admin, member, viewer）の二軸構成だ。この設計をPM Agentが `2026-03-12-role-permissions.md` に記録した。

```markdown
## 決定事項まとめ

### ロール構成
| 軸   | ロール         | 用途                                      |
| ---- | ------------- | ----------------------------------------- |
| 個人 | user (Free)   | 基本機能                                  |
| 個人 | pro (課金)    | 公開URL・PDF・Excel・AI補完増量・診断詳細 |
| 個人 | admin         | システム管理者。全機能+全組織アクセス     |
| 組織 | owner         | 組織設定・削除・メンバー管理              |
| 組織 | admin         | メンバー追加・削除                        |
| 組織 | member        | 検索・ブックマーク・ログ閲覧              |
| 組織 | viewer        | 閲覧専用。ブックマーク不可                |
```

さらに、機能アクセス権限マトリクスを4つの表（個人機能、組織機能、組織内ロール別、システム管理）で網羅的に定義した。Implement Agentはこのマトリクスを見ながら `feature-gate.ts` の実装と各APIのアクセス制御を正確に実装できた。

PM Agentがここまで詳細な権限マトリクスを出力したのは、権限設計は「後から追加」すると漏れが生じやすい領域だからだ。最初に全体像を定義し、Implement Agentに渡すことで、機能ごとにバラバラの実装にならず統一的なアクセス制御が実現した。

## 依存関係の可視化

PM Agentが出力するタスク分解には、必ず依存関係図を含めている。v0の例を示す。

```
#1 (DB setup)
  └─> #2 (CRUD API)
        ├─> #4 (登録フォーム) ──> #6 (詳細・編集) ──> #7 (削除)
        │     └──────────────────> #13 (抽出UI)
        ├─> #12 (抽出API) ──────> #13 (抽出UI)
        └─> #14 (認証)

#3 (ヘッダー)
  ├─> #4 (登録フォーム)
  ├─> #5 (一覧テーブル)
  └─> #9 (検索)

#5 (一覧テーブル)
  ├─> #8 (フィルター・ソート)
  ├─> #10 (ピン留め)
  └─> #11 (Topページ)
```

この図があることで、Implement Agentは「次に何を実装すべきか」を迷わない。#1（DB setup）が終わったら#2（CRUD API）と#3（ヘッダー）を並列に進められる。#4は#2と#3の両方に依存するから、両方が完了してから着手する。

v2ではさらにPhase分割で並列性を明示した。

```
Phase A（並列）: v2-1, v2-3, v2-7, v2-9
Phase B（依存）: v2-2, v2-4, v2-5, v2-8, v2-10
Phase C（統合）: v2-6
```

Phase Aの4つのissueは互いに依存しないため、worktreeを使って並列に実装できる。Phase Bの5つはPhase Aの各issueに依存する。Phase Cは全体を統合するissue。この構造化によって、Implement Agentの実行順序が明確になった。

## Tips

### PM Agentを効果的に使うコツ

**1. 入力は曖昧でいい**

PM Agentへの要望は「スキルシート機能を作りたい」程度でいい。PM AgentがCLAUDE.mdとdocs/decisions/の既存記録を読み、具体的なタスクに分解してくれる。

ただし、制約条件は明確に伝える。「コスト0で運用したい」「v0ではAI APIを使わない」のような制約がないと、PM Agentは最適と思われるアプローチ（この場合はAnthropic API呼び出し）を採用してしまう。DevDexでは実際に「長文用語抽出をAnthropic APIで実装する」というv0の初期案を、コスト制約を伝えて「辞書ベースのルールマッチング」に変更させた。

**2. 却下リストを必ず要求する**

PM Agentに「採用した方針だけ」を書かせると、後から「なぜ別の方法にしなかったのか」が分からなくなる。テンプレートに「却下した選択肢」セクションを含めることで、PM Agentは自動的に代替案とその却下理由を記録する。

**3. 見積もり精度は70%で十分**

PM Agentの工数見積もり（1h, 1.5h, 2hなど）は目安だ。実際にImplement Agentが実装すると、テストの追加やエッジケース対応で見積もりを超えることもある。しかし、見積もりがあるだけで「このissueは想定より大きいから分割すべきかもしれない」という判断ができる。

**4. 意思決定記録は「未来の自分への手紙」**

docs/decisions/は現在のPM Agent向けではなく、3か月後にコードを読む自分向けだ。「なぜRoute HandlersでServer Actionsは使わないのか」「なぜprofilesテーブルをauth.usersと別に作ったのか」。これらの判断理由は、コードだけ見ても分からない。PM Agentの記録が、コードの行間を埋めてくれる。
