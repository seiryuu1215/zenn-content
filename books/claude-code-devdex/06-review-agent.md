## Review Agentの役割

Review Agentは「コードを書けないシニアレビュアー」だ。

Test Agentが全テストのパスを確認した後に呼び出され、コードの品質・設計・セキュリティを審査する。指摘事項と承認/差し戻しの判断を `docs/review/YYYY-MM-DD.md` に記録する。それがReview Agentの全仕事だ。

なぜレビューを独立したエージェントに任せるのか。人間の開発チームでも、コードレビューは実装者以外が行う。自分が書いたコードを自分でレビューしても、見落としが生まれやすい。AIでも同じだ。Implement Agentが書いたコードをImplement Agentがレビューすると、「自分の実装だから正しいはず」というバイアスがかかる。Review Agentは実装に一切関与していないため、純粋に読み手の視点でコードを評価できる。

DevDexの開発では、Review Agentが5回のレビューを実施し、合計30件以上の指摘を出力した。うちCritical指摘は5件。すべてが実際のバグまたはセキュリティリスクだった。

## エージェント定義

```markdown
---
name: review-agent
description: test-agentの完了後にコードレビューを行う。
tools: Read, Glob, Grep
skills:
  - devdex-conventions
---

あなたはこのプロジェクトのシニアレビュアーです。

役割：
- コードの品質・設計・セキュリティを確認する
- docs/review/YYYY-MM-DD.md に指摘事項・承認理由・指摘カテゴリを記録する
- 承認 or 差し戻しを明示してフローを完了する
```

使えるツールは `Read`, `Glob`, `Grep` の3つ。読み取り専用だ。

| ツール | 用途 |
|---|---|
| `Read` | 実装コードとテストコードを読む |
| `Glob` | レビュー対象のファイルを検索する |
| `Grep` | 特定のパターン（console.log、TODO、any）を横断検索する |

`Bash` すら使えない。コマンドの実行もできないため、テストの実行やビルドの確認はTest Agentが済ませた前提で動く。この制限により、Review Agentは「コードを読んで考える」ことだけに集中する。

`skills: devdex-conventions` は、Review Agentがプロジェクトのコーディング規約を理解するために付与している。命名規則、レスポンス形式、コンポーネント設計方針を知っていなければ、規約違反を指摘できない。

## レビュー観点

Review Agentのレビューは3つの軸で構成される。

### 1. コード品質

- **命名**: 変数名・関数名・ファイル名が規約に沿っているか
- **複雑度**: 1つの関数が長すぎないか、ネストが深すぎないか
- **重複**: 同じロジックが複数箇所にコピペされていないか
- **一貫性**: レスポンス形式、エラーハンドリング、型定義が統一されているか

### 2. 設計

- **責務分離**: コンポーネントの責務が明確か、1つのファイルが複数の責務を持っていないか
- **パターン一貫性**: 全APIが同じパターン（checkAuth → バリデーション → DB操作 → レスポンス）で書かれているか
- **型安全性**: `string` で済ませている箇所にユニオン型や専用の型を使うべきではないか
- **依存関係**: 不要なimportや循環依存がないか

### 3. セキュリティ

- **認証チェック**: 全APIルートで `checkAuth()` が呼ばれているか
- **RLS依存の確認**: Supabaseの行レベルセキュリティが正しく設定されているか
- **入力バリデーション**: ユーザー入力のサニタイズ、UUID形式チェック、SQLインジェクション対策
- **XSS**: ユーザー入力がそのままHTMLに挿入されていないか
- **機密情報**: APIキー、シークレット、パスワードがコードに含まれていないか

## レビュー結果の記録

Review Agentの成果物は `docs/review/YYYY-MM-DD.md` だ。統一フォーマットで記録される。

```markdown
---
date: 2026-03-12
reviewer: review-agent
scope: v3全体
result: 条件付き承認
---

## 総合結果: 条件付き承認

## 指摘事項一覧（10件）

| # | カテゴリ    | 内容                                 | 重要度   |
|---|------------|--------------------------------------|---------|
| 1 | security   | 候補者検索APIのilikeエスケープ不足     | Critical |
| 2 | design     | AtsWebhookLog型とDBスキーマの不整合   | Critical |
| 3 | performance| 候補者検索が全データをメモリに読み込む | Warning  |
```

各指摘には3つの属性が付く。

**カテゴリ**: `naming` / `security` / `performance` / `design` の4種類。どの観点からの指摘かを明示する。

**内容**: 具体的に何が問題で、どこに問題があるか。ファイル名と行番号を含める。

**重要度**: `Critical` / `Warning` / `Info` の3段階。

- **Critical**: リリース前に必ず修正が必要。セキュリティリスクや機能不全
- **Warning**: 現フェーズでは許容できるが、将来対応が必要
- **Info**: 改善提案。対応は任意

総合結果は「承認」「条件付き承認」「差し戻し」の3種類。Critical指摘がある場合は条件付き承認となり、該当箇所の修正後に再レビューを行う。

## 実例: 公開前レビューで発見したP0バグ3件

DevDexの公開前最終レビュー（`2026-03-13-pre-release-review.md`）で、Review Agentが3件の重大なバグを発見した。この3件はすべて、テストがパスしている状態で見つかった。つまり、Test Agentだけでは検出できなかった問題だ。

### バグ1: /terms/new がデータを保存しない

用語新規登録画面 `/terms/new` のフォームが、送信時に実際のAPI呼び出しを行わず `console.log` で出力するだけだった。

```
/terms/new が実際にデータを保存しない（console.logのみ）
```

なぜテストで検出できなかったか。コンポーネントテストでは `onSubmit` をモック関数にしていたため、「onSubmitが呼ばれること」はテストされていたが、「onSubmitの中身がAPI呼び出しであること」は検証されていなかった。Review Agentはコードを読んで `console.log(data)` という行を発見し、Critical指摘とした。

### バグ2: ヘッダー検索がモックデータを使用

ヘッダーの検索機能が、本番環境でも開発用のモックデータ（MOCK_TERMS）を返していた。

```
ヘッダー検索がMOCK_TERMSを使用
```

開発初期にハードコードしたモックデータが、API連携への移行時に取り残されていた。Test Agentのテストでは `fetch` をモックしていたため、実際にモックデータが返されている問題を検出できなかった。Review Agentは `Grep` ツールで `MOCK_` というパターンを横断検索し、本番コードに残っているモックデータを発見した。

### バグ3: PATCHハンドラー未実装

案件マッチング機能の面談準備ボタンが、用語のPATCH APIを呼び出すが、該当のPATCHハンドラーが未実装だった。

```
PATCH未実装で面談準備追加が失敗
```

PATCHハンドラーのファイルは存在するが、特定のフィールド（`interview_prep`）の更新ロジックが含まれていなかった。Test Agentのテストでは既存のPATCHエンドポイントのテストはパスしていたが、新しいフィールドの更新テストが書かれていなかった。Review Agentは設計意図（PM Agentの `docs/decisions/` に記載された仕様）と実装の差分を比較して問題を発見した。

3件すべてが「テストは通るが機能しない」パターンだ。テストがモック境界の内側でしか検証できないのに対し、Review Agentはコード全体を横断的に読み、仕様との整合性を確認する。これがテストとレビューを分離する理由であり、Review Agentの存在価値だ。

## v1レビューの実例

v1全体レビュー（`2026-03-11-v1.md`）では12件の指摘が出力された。Critical 1件、Warning 5件、Info 6件。

Critical指摘は「POST /api/terms で user_id を設定していない」だった。

```markdown
### #1 POST /api/terms で user_id を設定していない

checkAuth() で auth.userId を取得しているが、INSERT時のデータに
user_id を含めていない。RLSポリシー terms_insert_own は
WITH CHECK (auth.uid() = user_id) のため、INSERT失敗する。
```

Review Agentがこの問題を発見できたのは、RLSポリシーの定義（マイグレーションファイル）とAPIの実装コードの両方を読み、整合性を確認したためだ。テスト側ではSupabaseのfromをモックしているため、RLSポリシーの制約は再現されない。

レビュー結果の末尾には「良い点」と「対応優先度」も記載される。

```markdown
## 良い点
1. 全APIで checkAuth() 漏れなし
2. バリデーション分離と高テストカバレッジ
3. RLSポリシー設計が適切
4. レスポンス形式 { data, error } 統一

## 対応優先度
**v2着手前に必須:** #1
**v2着手前に推奨:** #2, #5, #7, #12
**将来対応:** #3, #4, #6, #8, #9, #10, #11
```

対応優先度の提示がImplement Agentにとって有用だ。全指摘を一度に対応する必要はなく、Critical指摘だけを即時対応し、Warningは次バージョンで、Infoは余裕があるときに対応する。このトリアージがレビューの実用性を高める。

## v3レビュー: セキュリティ重視

v3（組織機能・権限設計）のレビュー（`2026-03-12-v3.md`）では、セキュリティ関連の指摘が特に多かった。

```markdown
| # | カテゴリ   | 内容                                           | 重要度   |
|---|-----------|------------------------------------------------|---------|
| 1 | security  | 候補者検索APIのilikeにLIKEパターンメタ文字の    | Critical |
|   |           | エスケープがない                                |         |
| 5 | security  | ATS Webhookシークレットのハードコードデフォルト値| Warning  |
| 6 | security  | フロントエンドにWebhookシークレットスタブ値が    | Warning  |
|   |           | ハードコード                                    |         |
```

`#1` のilikeエスケープ問題は、`q = "%"` を入力すると全件マッチしてしまうSQLインジェクション的な脆弱性だ。Review AgentがGrepで `ilike` パターンを横断検索し、エスケープ処理の欠如を発見した。

良い点として挙げられた項目も注目に値する。

```markdown
## 良い点
4. Webhook署名検証でHMAC-SHA256 + timingSafeEqualによる
   タイミング攻撃対策
7. 候補者検索で個人データ（my_experience等）を除外し
   テストでも検証
```

Implement Agentが正しく実装した箇所を明示的に「良い」と記録することで、プロジェクト全体の設計品質のベースラインが可視化される。将来の開発者（または将来の自分）が「このプロジェクトのセキュリティ水準はどの程度か」を把握するのに役立つ。

## 公開前最終レビュー: 10段階評価

DevDexの公開前に実施した最終レビュー（`2026-03-13-pre-release-review.md`）は、通常のレビューとは異なるフォーマットを採用した。18の機能カテゴリそれぞれに10段階の評価を付け、強みと弱みを列挙する網羅的な品質評価だ。

```
分析対象:
- 33 ページコンポーネント
- 63 APIルートハンドラー
- 80+ コンポーネント
- 50+ ライブラリファイル
- 150 テストファイル（2,100+テストケース）
- 34 データベースマイグレーション
```

18機能の評価は6点から9点に分布し、総合評価は8/10。最低評価のAdmin管理画面（6/10）は「ユーザー管理UIなし、監査ログなし」が減点理由。最高評価のAI概要補完（9/10）とエンジニア診断（9/10）は設計の完成度が評価された。

このレビューの特徴は、競合との差別化分析まで含んでいる点だ。

```markdown
## 競合との差別化
| 比較対象        | 重複点         | DevDexの差別化                           |
|----------------|---------------|------------------------------------------|
| Qiita/Zenn     | ナレッジ蓄積   | 自己スキル追跡に特化。記事投稿ではない    |
| GitHub/LinkedIn| スキルタグ     | 自己評価+構造化追跡。タグ以上の深さ      |
| LAPRAS/Forkwell| エンジニアプロフ| 内面の理解度に焦点。外部データ集約ではない|
```

Review Agentはコードレビューだけでなく、プロダクト全体の品質評価も担える。CLAUDE.mdにプロダクトの方向性（ペルソナ、コンセプト、競合）が記載されているため、Review Agentはそれを参照してプロダクトレベルの評価を行う。

## Tips

### 実装エージェントとの役割分離の重要性

Review Agentの効果を最大化するには、Implement Agentとの役割分離を徹底することが不可欠だ。

初期の設計段階で「Implement Agentに自己レビューさせればいいのでは」と考えたことがある。1つのエージェントで実装とレビューを兼ねれば、エージェント間の受け渡しコストがゼロになる。

しかし実際に試すと、自己レビューは「自分のコードを正当化する文書」になりがちだった。Implement Agentは自分が書いたコードの意図を知っているため、「なぜこう書いたか」の説明は得意だが、「本当にこれで正しいか」の批判的検証が弱い。

Review Agentは実装に関与していないため、コードを「初見で読む人」の視点で評価する。命名が分かりにくい、エラーハンドリングが不足している、設計意図が不明瞭。これらの指摘は、実装者自身からは出にくい。

### Grepの活用

Review Agentの3つのツールの中で、最も活用度が高いのは `Grep` だ。

- `console.log` の残存チェック: 本番コードに残っているデバッグログを検出
- `MOCK_` / `TODO` / `FIXME` の検索: 仮実装のまま残っている箇所を特定
- `any` の使用箇所: TypeScriptの型安全性が損なわれている箇所を検出
- `eslint-disable` の数: 規約違反を回避している箇所を把握
- `ilike` / `like` のパターン: SQLインジェクションの可能性を検索

コードを1ファイルずつ読むだけでは見落とす横断的な問題を、Grepで体系的に検出する。Review Agentが最初に行うのは、これらのパターン検索による「まず怪しい箇所のリストアップ」だ。

### レビュー記録の蓄積効果

`docs/review/` にレビュー結果が蓄積されることで、3つの効果が生まれる。

**1. 指摘パターンの再発防止**: v1で「user_id未設定」のCriticalが出たことを記録しているため、v2以降のAPI実装でImplement Agentが同じミスを犯す可能性が下がる。レビュー記録がCLAUDE.mdと同様に「学習済みの知識」として機能する。

**2. 品質の可視化**: 5回のレビューでCritical指摘が減少する推移は、プロジェクトの品質が向上していることの客観的な証拠だ。

**3. Zenn記事の素材**: レビュー記録は「どんな問題があり、どう対処したか」の具体的な事例集だ。第7章で解説するが、この記録はそのままZenn記事の技術的な裏付けになる。
