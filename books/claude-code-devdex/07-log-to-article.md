## 3層のログ蓄積戦略

DevDexの開発では、情報を3つの層で蓄積する設計を採用した。

```
Layer 1: PR単位の日記      → docs/diary/YYYY-MM-DD.md
Layer 2: 意思決定記録      → docs/decisions/YYYY-MM-DD-[topic].md
Layer 3: 自動ログ          → docs/logs/YYYY-MM-DD.jsonl
```

この3層構造の狙いは「粒度の異なる情報を、それぞれ最適な形式で残す」ことだ。

**Layer 1（日記）** は最も粗い粒度。1日の作業を要約した人間可読なMarkdown。「何をしたか」「テスト数はいくつか」「次に何をするか」が書いてある。

**Layer 2（意思決定記録）** は設計判断に特化した中粒度の記録。「なぜServer Actionsを使わなかったか」「なぜ権限設計を二軸にしたか」のような、コードだけでは分からない判断の理由が書いてある。

**Layer 3（自動ログ）** は最も細かい粒度。全ツール実行をJSON Lines形式で記録する機械可読なログ。「何時何分にどのファイルを編集したか」「tscの実行結果は何だったか」が全て残る。

この3層は相互補完の関係にある。Layer 3の自動ログだけでは膨大すぎて人間が読めない。Layer 1の日記だけでは設計判断の「なぜ」が抜ける。Layer 2の意思決定記録だけでは日々の進捗が見えない。3つを組み合わせることで、開発の全体像と詳細の両方が残る。

## Diary Agentの役割

Diary Agentは6体のサブエージェントの中で、唯一「開発の成果物を直接生産しない」エージェントだ。

```markdown
---
name: diary-agent
description: 1日の作業終了時に呼び出す。
tools: Read, Glob
---

あなたはこのプロジェクトの記録担当です。実装はしません。

役割：
- docs/decisions/ と docs/review/ と当日のテスト結果と
  docs/logs/YYYY-MM-DD.jsonl を参照する
- docs/diary/YYYY-MM-DD.md に統合してまとめる
- 末尾にZenn投稿用サマリー（200字）を追記する
```

使えるツールは `Read` と `Glob` だけ。コードの読み取りとファイル検索しかできない。実装もテストもレビューもしない。ただひたすら「他のエージェントの成果物を読んで、日記にまとめる」だけだ。

### PR単位のミニエントリ

Diary Agentの出力は2段階で構成される。

第1段階は **PR単位のミニエントリ**。PRがマージされるたびに、以下の形式でエントリが追記される。

```markdown
### PR #149: AI概要補完機能を実装する

- **変更**: 5ファイル。ai-overview API新規、
  TermFormにAI補完ボタン追加、ai_overviewsテーブル新規
- **設計判断**: 辞書→共有キャッシュ→Anthropic API(Haiku)の
  3段階フォールバック。1用語につきAPI1回のみでコスト最適化。
  1日10回制限
- **テスト**: 1279件パス（+10件追加）
```

3行で1つのPRの全体像が把握できる。「変更」で何を触ったか、「設計判断」でなぜそうしたか、「テスト」でどの程度の品質が担保されたか。この3つの情報があれば、3か月後に「PR #149って何だったっけ」と振り返るときに十分だ。

### セッション終了時のサマリー

第2段階は **1日のサマリー**。セッション終了時に `/diary` コマンドで呼び出し、その日のミニエントリ群を統合する。

```markdown
## 1日のサマリー

- 完了PR: #148, #149, #151
- テスト総数: 1284件（+15件追加）
- レビュー結果: 2件実施
  - v3全体レビュー: 条件付き承認
    （Critical 2件 → 対応済み / Warning 4件 / Info 4件）
  - UX改善レビュー: 承認
    （Warning 3件 / Info 5件）

## 意思決定サマリー

### 1. AI概要補完機能（Issue #43）
用語登録時のoverview入力負担を軽減するため、
3段階フォールバック方式を採用した。
- 辞書（117件）→ 共有キャッシュ（ai_overviews）
  → Anthropic API（Claude Haiku）
- 全ユーザー共有キャッシュにより、同じ用語は1回のAPI呼び出し
- 却下案: ユーザー別キャッシュ（同一用語で重複コスト）

## 課題・気づき

- v3レビューのWarning残件は本番化前に対応が必要
- 日記生成をPR単位に変更。セッション終了時の一括生成は
  情報が膨大になりすぎる問題を解消
```

サマリーの中で特に価値が高いのは「意思決定サマリー」セクションだ。Diary Agentは `docs/decisions/` を読んでその日の設計判断を要約する。PM Agentが書いた意思決定記録をDiary Agentが咀嚼し、日記の文脈で再構成する。これにより、意思決定記録を直接読まなくても日記から設計判断の概要が把握できる。

### Zenn投稿用サマリー

日記の末尾にはZenn投稿用のサマリー（200字以内）が自動追記される。

```markdown
### Zenn投稿用サマリー（200字）

DevDex開発12日目。AI概要補完機能をリリースした。
内蔵辞書117件→共有キャッシュ→Claude Haiku APIの
3段階フォールバックで、コストを最小化しつつ用語登録の
入力負担を軽減する設計。v3レビュー指摘7件を一括対応し
TypeAvatarのセキュリティ強化や候補者検索の実データ取得
を実装。テスト1284件。
```

200字に圧縮することで、Zenn記事やSNS投稿にそのまま使える素材になる。Diary Agentは日記の全内容を200字に要約する能力が求められるため、「何が最も重要で、何を省略できるか」の判断を自動的に行う。

## 意思決定記録の設計

`docs/decisions/` はPM Agentの成果物だが、Diary Agentと並んでZenn記事の最も重要な素材源だ。

DevDexでは25件の意思決定記録が残っている。

| ファイル名 | テーマ |
|---|---|
| `2026-03-11-v0-task-breakdown.md` | v0のタスク分解（14 issue） |
| `2026-03-11-extract-approach.md` | 用語抽出のアプローチ選定 |
| `2026-03-11-dark-mode.md` | ダークモード対応 |
| `2026-03-11-v1-task-breakdown.md` | v1のタスク分解（7 issue） |
| `2026-03-11-v2-task-breakdown.md` | v2のタスク分解（10 issue） |
| `2026-03-12-ai-overview.md` | AI概要補完機能の設計 |
| `2026-03-12-role-permissions.md` | 権限・ロール設計 |
| `2026-03-12-docs-portfolio-zenn.md` | ドキュメント・書籍計画 |
| ... | （計20件） |

各記録には「背景・課題」「採用した方針」「却下した選択肢」「タスク分解」の4セクションがある。この中で記事の素材として最も使えるのは「却下した選択肢」だ。

なぜか。技術記事を読む人が最も知りたいのは「なぜその技術を選んだか」ではなく「なぜ他の選択肢を選ばなかったか」だからだ。「Server Actionsを使わなかった理由」「Firebase でなくSupabaseを選んだ理由」「認証を最後に実装した理由」。これらの「不採用の理由」は、読者が自分のプロジェクトで同じ判断をするときの参考になる。

DevDexの `docs/decisions/2026-03-11-v0-task-breakdown.md` から引用する。

```markdown
## 却下した選択肢

| 選択肢               | 却下理由                              |
|---------------------|---------------------------------------|
| Server ActionsでCRUD | フォームと密結合しすぎる。API境界が     |
|                     | 不明確。将来モバイル対応時に再実装必要  |
| 認証を最初に実装     | 全機能の開発がブロックされる。まず      |
|                     | ローカルで動くものを作り最後に認証      |
```

この「却下した選択肢」テーブルが、そのまま技術選定記事の1セクションになる。

## レビュー記録からの素材

`docs/review/` のレビュー記録も記事の素材として活用する。

v1レビューの「POST /api/terms で user_id を設定していない」というCritical指摘は、「AIコードレビューが実際にバグを見つけた事例」としてそのまま記事になる。テストが全パスしている状態で、レビューが見つけたバグ。テストの限界とレビューの必要性を示す具体例だ。

v3レビューの「ilikeエスケープ不足」は、セキュリティ記事の素材になる。`q = "%"` で全件マッチするSQLインジェクション的な脆弱性を、AIレビュアーが横断検索で発見した事例。

レビュー記録の「良い点」セクションも使える。「Webhook署名検証でHMAC-SHA256 + timingSafeEqualによるタイミング攻撃対策」は、セキュリティ設計の好例としてそのまま引用できる。

## 3層ログからZenn記事への変換パイプライン

DevDexの3層ログから記事への変換は、3つのルートがある。

### ルート1: 日記 → 「開発日誌」シリーズ

`docs/diary/YYYY-MM-DD.md` の日記は、Zenn記事の「開発日誌」シリーズの素材になる。1日の日記をそのまま投稿するのではなく、複数日の日記を統合して「1週間のふりかえり」や「v0からv1への進化」のようなストーリーに再構成する。

日記のミニエントリは時系列で並んでいるため、「機能がどう積み上がっていったか」のストーリーが自然に生まれる。v0の403テストが2,111テストに成長する過程、Critical指摘が減っていく過程。これらの推移は、読み物としても面白い。

### ルート2: 意思決定記録 → 「技術選定・設計編」

`docs/decisions/` の意思決定記録は、技術選定・設計判断の記事に直結する。

- 「Server Actions vs Route Handlers、どちらを選ぶか」→ v0タスク分解の却下リスト
- 「AI概要補完のコスト最適化設計」→ ai-overview.md の3段階フォールバック
- 「個人ロールと組織ロールの二軸設計」→ role-permissions.md の権限マトリクス

意思決定記録のフォーマットが「背景・方針・却下・タスク」で統一されているため、記事のアウトラインもほぼそのまま使える。「背景」がリード文、「方針」が本題、「却下」が代替案の比較、「タスク」が実装の具体例。

### ルート3: PRの設計判断セクション → 実装判断の根拠

PRのテンプレートには「設計判断」セクションが含まれている。

```markdown
## 設計判断

- AI概要補完は辞書→共有キャッシュ→Anthropic APIの
  3段階フォールバックを採用。辞書・キャッシュヒットは
  レート制限にカウントしないことでFree枠の10回制限でも
  実用的に使えるようにした
- 却下案: ユーザー別キャッシュ（同一用語で重複コスト）
```

このセクションは、docs/decisions/ よりもさらに具体的な「実装時の判断」が記録されている。「コードレベルでなぜこう書いたか」の根拠だ。技術記事で実装の詳細を説明するときに、この記録が裏付けになる。

## この書籍自体がその導線の産物

ここまで読んで気づいた方もいるかもしれない。この書籍自体が、DevDexの3層ログから生まれた産物だ。

PM Agentが `docs/decisions/2026-03-12-docs-portfolio-zenn.md` でZenn書籍の計画を立てたとき、書籍の素材として参照すべきファイルとして `docs/decisions/` と `docs/diary/` を明示した。

```markdown
### Phase G: Zenn書籍（10 issue, 約18h）
タイトル案: 「Claude Code サブエージェント6体制で
            2,111テストのWebアプリを作った方法」
```

この書籍の各章の内容は、DevDexの開発過程で蓄積されたログから構成されている。

- **第1章（DevDexとは何か）**: CLAUDE.mdのプロダクト概要から
- **第2章（サブエージェント設計）**: `.claude/agents/` の定義ファイルと `.claude/settings.json` から
- **第3章（PM Agent）**: `docs/decisions/` の実例から
- **第4章（Implement Agent）**: 実装コードのパターンと `.claude/skills/` から
- **第5章（Test Agent）**: テストファイル94本の構造とパターンから
- **第6章（Review Agent）**: `docs/review/` の5回分のレビュー記録から
- **第7章（この章）**: 3層ログの設計と `docs/diary/` の実例から

つまり、DevDexの開発中に「記事を書こう」と意識していたわけではない。サブエージェント体制の副産物としてログが蓄積され、そのログが自然に記事の素材になった。

これが3層ログ蓄積戦略の本質だ。「記事を書くために記録する」のではなく、「開発のために記録したものが記事になる」。PM Agentは実装のために設計判断を記録し、Diary Agentはプロジェクト管理のために日記を生成し、Review Agentは品質保証のためにレビュー結果を残す。それぞれのエージェントが自分の本来の仕事をした結果として、副産物としてのログが蓄積される。

## Tips

### 日記の粒度制御

初期の設計では、1日の終わりにDiary Agentが全PR分を一括で日記化する方式だった。しかし、1日に5〜6件のPRがマージされると情報量が膨大になり、要約の品質が下がった。

対策として、PR単位のミニエントリ方式に変更した。PRがマージされるたびに3行のミニエントリを追記し、セッション終了時にはそれらを統合してサマリーを生成する。この2段階方式により、情報の欠落なく日記の品質を維持できた。

### 意思決定記録のテンプレート遵守

PM Agentに自由にドキュメントを書かせると、ファイルごとにフォーマットがバラバラになる。`docs/decisions/TEMPLATE.md` を用意し、PM Agentの定義に「テンプレートに従って記録する」と明記することで、20件すべてが同一フォーマットで統一された。

フォーマットの統一は、後の検索と引用を劇的に楽にする。「却下した選択肢を全件抽出する」「タスク分解の見積もりを集計する」のような横断的な分析が、フォーマットが揃っているからこそ可能になる。

### ログの「捨てる」判断

3層すべてのログを永久保持する必要はない。Layer 3の自動ログ（JSONL）は量が膨大になるため、月次でアーカイブまたは削除してよい。Layer 1の日記とLayer 2の意思決定記録は永久保持する。

この判断は「何が記事の素材になるか」で決める。JSONL形式の生ログが記事になることはほぼない。しかし日記と意思決定記録は、開発が終わった後でも参照する価値がある。

### 書籍化のタイミング

DevDexの書籍化をv3完了後に行ったのは、全体像が確定してから俯瞰的な記述ができるためだ。開発途中に書き始めると、後から構成を変更する手間が大きい。

ただし、ログの蓄積は開発と並行して行う。書籍化は事後的だが、素材の蓄積は同時進行。この非同期な設計が、「開発を止めずに記事の素材を貯める」仕組みの核心だ。
