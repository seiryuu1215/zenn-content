## AI駆動開発で失敗したこと

### 1. コンテキストウィンドウの消費

6体のエージェントを同時に動かすと、コンテキストウィンドウの消費が激しい。長い会話になるほど前の決定を忘れ、矛盾した実装をすることがあった。

**対策**: `docs/decisions/` に24個の設計判断書を残し、エージェントの「記憶」として機能させた。1 issue = 1 branch = 1 PRの粒度ルールで、各タスクのスコープを小さく保った。

### 2. テストのモック設計の不統一

初期はテストごとにモックパターンがバラバラだった。Supabaseクライアントのチェーンモック（`.from().select().eq().single()`）が特に混乱の原因になった。

**対策**: CLAUDE.mdにモックテンプレートを明記。Test Agentが常に同じパターンで書けるようにした。

### 3. RLSポリシーの穴

AIが生成したRLSポリシーが「動くけど穴がある」状態になりやすい。特に中間テーブル（term_relations）のポリシーで、サブクエリの条件が不十分なケースがあった。

**対策**: Review Agentにセキュリティレビューを明示的に指示。RLSポリシーだけは全件手動確認した。

### 4. Vercel環境変数の罠

Supabaseの秘密鍵をVercelダッシュボードから設定すると、改行文字(`\n`)が文字列としてそのまま保存される。デプロイ後にJWT検証が失敗して数時間詰まった。

**対策**: `printf` コマンドで環境変数を設定。Vercel CLIやAPIを使う場合も同様の注意が必要。

### 5. 曖昧な指示の代償

「いい感じにして」「UIを改善して」のような曖昧な指示は、期待と全く違う結果になることが多かった。特にデザイン面での「いい感じ」は人によって定義が違う。

**対策**: ファイルパス・型・既存パターンへの参照を明示。「`src/app/api/terms/route.ts` と同じパターンで」のように具体化するだけで精度が劇的に上がった。

## 失敗から得たルール

1. **設計判断は必ず記録する** -- AIは忘れるが、ファイルは忘れない
2. **テンプレートを先に作る** -- 統一パターンがあればAIの出力がブレない
3. **セキュリティは信じない** -- AIの出力を信じすぎず、重要な部分は手動確認
4. **具体的に指示する** -- 曖昧さはコストに直結する
5. **タスクは小さく切る** -- コンテキストウィンドウは有限
