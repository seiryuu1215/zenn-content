## AI駆動開発で失敗したこと

### 1. コンテキストウィンドウの消費

6体のエージェントを同時に動かすと、コンテキストウィンドウの消費が激しい。長い会話になるほど前の決定を忘れ、矛盾した実装をすることがあった。

**対策**: `docs/decisions/` に24個の設計判断書を残し、エージェントの「記憶」として機能させた。1 issue = 1 branch = 1 PRの粒度ルールで、各タスクのスコープを小さく保った。

### 2. テストのモック設計の不統一

初期はテストごとにモックパターンがバラバラだった。Supabaseクライアントのチェーンモック（`.from().select().eq().single()`）が特に混乱の原因になった。

**対策**: CLAUDE.mdにモックテンプレートを明記。Test Agentが常に同じパターンで書けるようにした。

### 3. RLSポリシーの穴

AIが生成したRLSポリシーが「動くけど穴がある」状態になりやすい。特に中間テーブル（term_relations）のポリシーで、サブクエリの条件が不十分なケースがあった。

**対策**: Review Agentにセキュリティレビューを明示的に指示。RLSポリシーだけは全件手動確認した。

### 4. Vercel環境変数の罠

Supabaseの秘密鍵をVercelダッシュボードから設定すると、改行文字(`\n`)が文字列としてそのまま保存される。デプロイ後にJWT検証が失敗して数時間詰まった。

**対策**: `printf` コマンドで環境変数を設定。Vercel CLIやAPIを使う場合も同様の注意が必要。

### 5. 曖昧な指示の代償

「いい感じにして」「UIを改善して」のような曖昧な指示は、期待と全く違う結果になることが多かった。特にデザイン面での「いい感じ」は人によって定義が違う。

**対策**: ファイルパス・型・既存パターンへの参照を明示。「`src/app/api/terms/route.ts` と同じパターンで」のように具体化するだけで精度が劇的に上がった。

### 6. 収益化設計のタイミングが遅すぎた

Stripe連携・Free/Proの線引きがDay 4まで後回しになった。機能を先に作り、後からFeature Gateを全箇所に挿入する手戻りが発生。FeatureGateコンポーネントの呼び出し箇所が増えるたびに、既存のテストも書き直す羽目になった。

**対策**: PM AgentのDay 1要件整理に「収益化設計（Free/Pro境界、課金フロー）」を必須項目として追加。機能を作る前に「これはFreeかProか」を決めておく。

### 7. 外部サービスの事前準備不足

ドメイン取得（devdex.dev）、Anthropic APIキーのチャージ、Stripe本番キー発行。これらの外部手続きで想定外の待ち時間が発生した。特にAnthropicのクレジット未チャージは、本番でAI機能が全滅する事態を招いた。

**対策**: Day 1に「外部サービス事前チェックリスト」を作成。アカウント作成・キー発行・課金設定・ドメイン取得を開発開始前に済ませる。

### 8. ブランドアセットの要件定義漏れ

コード凍結後にファビコンがNext.jsデフォルトのままだと発覚。OGP画像やapple-touch-iconも含め、ブランドアセットがローンチ前チェックリストに入っていなかった。目標を「コード凍結」と設定していたのに、こういう基本的な項目が漏れるのは悔しい。

**対策**: ローンチ前チェックリストにブランドアセット項目（ファビコン・OGP画像・apple-touch-icon・PWAアイコン）を追加。

### 9. Issueのメタデータ管理が雑だった

GitHub Issueのlabels・milestone・assignee・linked issuesの設定が不十分で、振り返り時に「いつ何をやったか」の追跡が困難だった。543件のPRがあるのに、メタデータで検索・分類できない。

**対策**: サブエージェントのIssue作成時にlabels・milestone設定を必須化する。PM Agentの定義に組み込む。

### 10. マイグレーション番号の命名規則違反

`0291`、`0311`、`0312`のような番号付けでマイグレーション履歴の不整合が発生。本番適用時にrepair作業が必要になり、数時間を消費した。

**対策**: マイグレーション番号は厳密な連番を守る。途中挿入が必要な場合はスキーマ統合で対応。

## 失敗から得たルール

1. **設計判断は必ず記録する** -- AIは忘れるが、ファイルは忘れない
2. **テンプレートを先に作る** -- 統一パターンがあればAIの出力がブレない
3. **セキュリティは信じない** -- AIの出力を信じすぎず、重要な部分は手動確認
4. **具体的に指示する** -- 曖昧さはコストに直結する
5. **タスクは小さく切る** -- コンテキストウィンドウは有限
6. **収益化設計は最初にやる** -- 後から入れると手戻りが大きい
7. **外部サービスは先に準備する** -- APIキー・ドメイン・課金は開発の前に
8. **チェックリストを信じる** -- 「当たり前」だと思うことほど漏れる
9. **メタデータは未来の自分への投資** -- 雑にすると振り返りができない
10. **命名規則は例外を作らない** -- 1つの例外が連鎖的に問題を起こす
