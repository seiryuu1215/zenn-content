## 数値で見る開発成果

DevDexの開発を通じた最終的な数値を総括する。

| 指標 | 数値 |
|---|---|
| APIエンドポイント | 63本 |
| ページ数 | 33画面 |
| ユニットテスト | 2,100+件（150ファイル） |
| DBマイグレーション | 35本 |
| PR（マージ済み） | 543+件 |
| サブエージェント | 6体（PM, Implement, Test, Review, Diary, Biz） |
| 意思決定記録 | 20件超（docs/decisions/） |
| レビュー記録 | 5回（docs/review/） |
| 用語辞書（プリセット） | 117件 |

| 個人ロール | 3種類（user, pro, admin） |
| 組織ロール | 4種類（owner, admin, member, viewer） |

### フェーズ別の進捗

開発はv0からv3の4フェーズで段階的に進めた。

| フェーズ | issue数 | テスト数 | 主な成果 |
|---|---|---|---|
| v0（MVP） | 14 | 403 | CRUD API、用語一覧、検索、ピン留め、認証 |
| v1（AI+連携） | 7 | 496 | AI概要補完、マスターデータ、関連用語 |
| v2（公開・共有） | 10 | 1,025 | 診断、スキルシート、公開プロフィール |
| v3（企業向け） | 12 | 1,316 | 組織機能、Feature Gate、Admin |
| 公開準備 | 20+ | 1,407 | LP、オンボーディング、診断デモグラフィック、Stripe連携、SEO/OGP |
| 機能追加 | 30+ | 1,813 | ダッシュボード改善、バグ修正、UI改善 |
| 品質強化 | 15+ | 2,100+ | Pro限定機能、AI機能拡充、テスト網羅性向上 |

v0からv1にかけてはテスト数の増加が緩やかだが、v2からv3で急増している。権限・ロールの組み合わせテスト（3個人ロール x 4組織ロール x 11機能ゲート）がテスト数を押し上げた。

### 1日あたりの生産性

v0のMVP（14 issue）を実働3〜4日で完了した。1日あたり3〜5個のPRがマージされるペースだ。1 issueあたりの平均実装時間は1〜2時間。PM Agentのタスク分解で「1 issue = 1〜2時間」ルールを設定した効果がここに表れている。

サブエージェントフロー1周（PM → 実装 → テスト → レビュー）は、小さなissueなら30分、大きなissueでも2時間以内に完了した。1つの機能を実装してテストしてレビューしてマージするまでが、1セッションで完結する。

## Claude Codeの貢献度

### 生産性

サブエージェント6体制の最大の効果は「人間がやるべきこと」の最小化だ。

人間（自分）の作業は3つだけ。

1. **方向性を決める**: 「次にスキルシート機能を作りたい」のような要望を伝える
2. **判断を承認する**: PM Agentのタスク分解を確認し、必要なら修正を指示する
3. **最終確認**: 動作確認とデプロイ

コードを1行も書かない。テストも書かない。レビューもしない。それでいて、63本のAPIが統一されたパターンで実装され、2,100+件のテストが全パスし、5回のレビューでCritical指摘は全件対処された。

手動でこの規模のアプリを作った場合の工数を見積もると、少なくとも3〜4倍の時間がかかるだろう。特にテストの2,100+件を手動で書く作業と、全APIの認証チェック・バリデーション・エラーハンドリングを統一的に実装する作業は、AI無しでは数週間かかる。

### 品質

サブエージェント体制がもたらした品質上の効果を3つ挙げる。

**1. パターンの一貫性**。63本のAPIがすべて同一のパターン（checkAuth → バリデーション → DB操作 → `{ data, error }` レスポンス）で実装されている。人間が63本のAPIを書くと、途中で微妙にパターンが変わることがある。Implement AgentはCLAUDE.mdとスキルファイルを毎回参照するため、ブレない。

**2. テストの網羅性**。Test Agentが書くテストは「全API x 認証テスト + バリデーションテスト + 正常系テスト」を機械的に網羅する。人間なら「このAPIはシンプルだからテスト少なくていいか」と手を抜きがちだが、Test Agentに手抜きの概念はない。2,100+件のテストがその証拠だ。

**3. レビューの客観性**。Review AgentはGrepで横断的にパターン検索し、`console.log` の残存、`MOCK_` データの取り残し、`ilike` のエスケープ不足を発見した。これらは人間のレビューでも見落としやすいパターンだ。

### 一貫性

全コミットが `type: 日本語の説明` 形式で統一された。543+件のPRすべてが「概要・変更内容・設計判断・確認方法」のテンプレートに沿っている。20件超の意思決定記録すべてが「背景・方針・却下・タスク」のフォーマットで書かれている。

この一貫性は人間の開発チームでは達成困難だ。チームメンバーが増えるほどフォーマットのばらつきが大きくなる。AIエージェントはフォーマットを「忘れない」ため、プロジェクト全体で一貫性が維持される。

## AI駆動開発の課題と学び

DevDexの開発はうまくいった部分が多いが、課題も存在した。

### 課題1: コンテキスト制限

Claude Codeのコンテキストウィンドウは有限だ。大きなissue（特にスキルシート機能の7 issue一括実装）では、実装の後半でコンテキストが圧縮され、前半の設計意図が薄れることがあった。

対策として `Compact Instructions` を設定し、圧縮時に保持すべき情報を指定した。さらに、1 issueの粒度を「1〜2時間」に厳守することで、コンテキストが溢れる前にissueを完了させるリズムを作った。

根本的な解決は「ファイルシステム経由の情報伝達」だ。`docs/decisions/` に書かれた設計判断は、コンテキストが圧縮されても消えない。エージェントは必要に応じてファイルを `Read` で再読み込みできる。

### 課題2: エージェントの役割逸脱

初期にImplement Agentがテストも一緒に書いてしまうパターンが頻発した。実装者がテストを書くと、「自分のコードが正しい前提」でテストを組み立てるため品質が低い。

対策として、エージェント定義に「実装完了後 test-agent に引き渡す」と明記し、Test Agentのツール制限（Read, Bash, Globのみ）で実装コードの編集を防いだ。ツールの制限が、役割の境界を強制する。

もう1つの逸脱パターンが、PM Agentの「タスク分解が粗すぎる」問題だ。「スキルシート機能を実装する」のような大きなタスクだと、Implement Agentが途中で方向を見失う。PM Agentには「1 issue = 1〜2時間で終わる粒度」をルールとして課し、大きな機能は必ず複数issueに分解させた。

### 課題3: モックの限界

Test Agentが書くテストはSupabaseクライアントと認証関数をモックしている。このモックが「テストは通るが実際には動かない」問題を生んだ。

Review Agentが公開前レビューで発見した3件のP0バグ（/terms/newがデータを保存しない、ヘッダー検索がモックデータを使用、PATCHハンドラー未実装）はすべて、モック境界の外側で起きた問題だった。テストは `onSubmit` をモック関数にしていたため、「onSubmitが呼ばれること」はテストされていたが、「onSubmitの中身がAPI呼び出しであること」は検証されていなかった。

この課題の対策は2つ。Review Agentによるコード横断レビューと、E2Eテスト（Playwright）の導入だ。DevDexではPlaywrightの基盤構築まで完了しており、今後シナリオテストを追加する予定だ。

### 課題4: 承認フローの判断

サブエージェントフローでは「PM Agentの判断を人間が承認する」ステップがある。PM Agentの出力が優れている場合、この承認はスタンプラリーになりがちだ。一方で、PM Agentの判断を鵜呑みにすると、プロダクトの方向性がAI任せになる。

DevDexでは、PM Agentの出力を「採用」「修正して採用」「却下して再生成」の3つの選択肢で判断した。特にv3の権限設計では、PM Agentの初期案に「viewerロール」が含まれておらず、手動で追加を指示した。AI駆動開発でも、ドメイン知識を持つ人間の判断は不可欠だ。

### 学び: 情報のドキュメント化が全てを支える

DevDex開発の最大の学びは「ドキュメント化こそがAI駆動開発の生命線」だということだ。

CLAUDE.mdがなければ、エージェントはプロジェクトの方針を知らない。`docs/decisions/` がなければ、設計判断の理由が失われる。MEMORY.mdがなければ、セッション間の連続性が途切れる。`docs/review/` がなければ、過去の指摘パターンから学べない。

人間のチーム開発でも「ドキュメントを書こう」と言われるが、なかなか実行されない。AI駆動開発では、ドキュメントがないとエージェントが文字通り機能しないため、ドキュメント化が「やるべきこと」から「やらざるを得ないこと」に変わる。

## SubAgentパターンの応用可能性

DevDexで確立したサブエージェント6体制は、他のプロジェクトにもそのまま適用できる。

### 最小構成: 3体制

PM Agent + Implement Agent + Test Agentの3体制が最小構成だ。Review AgentとDiary Agentは省略可能。3体制でも「設計と実装の分離」と「実装とテストの分離」という2つの品質ゲートが機能する。

### フロントエンド特化: 4体制

デザインAgentを追加した4体制。PM Agent → Design Agent（UIモック生成）→ Implement Agent → Test Agent。Design AgentにはFigmaやv0のようなツール連携を持たせ、UIの設計をImplement Agentに渡す。

### データ分析: 3体制

分析Agent + 実装Agent + 検証Agent。分析Agentがデータの傾向を把握し仮説を立て、実装Agentがクエリやダッシュボードを構築し、検証Agentが結果の妥当性を確認する。

共通する原則は「役割を分離し、ツール制限で境界を強制する」ことだ。1つのエージェントに全ての役割を持たせると、コンテキスト汚染と品質低下が起きる。最低でも「設計する人」と「作る人」と「確認する人」を分けることで、品質ゲートが機能する。

## 今後の展望

### Expo iOS対応

DevDexのAPIはRoute Handlersで構築しているため、モバイルアプリからもそのまま呼び出せる。PM Agentが初期のv0タスク分解で「Server Actionsを使わない」と判断したのは、この将来のモバイル対応を見据えてのことだった。

### E2Eテスト

Playwright + CI/CD基盤は構築済み。今後、主要ユースケース（用語登録 → 編集 → 検索 → ピン留め → AI補完 → スキルシート）のE2Eシナリオを追加する。モックの限界を補完する手段として、E2Eテストは不可欠だ。

### 収益化

Feature Gateの実装は完了しており、Stripeなどの決済連携を追加すればPro課金が開始できる状態だ。toC（エンジニア向けPro）とtoB（企業向け組織プラン）の二軸で、まずは小規模に開始する。

### コミュニティ

DevDexの最終目標は「エンジニアの理解度を可視化するプラットフォーム」だ。個人ツールから始まり、公開プロフィールで他者と共有し、組織機能で企業と繋がる。この段階的な拡張は、アーキテクチャの設計段階から意図していた。

## 読者へのメッセージ

この書籍では、Claude Codeのサブエージェント6体制でDevDexを開発した全工程を記録した。

PM Agentが20件超の意思決定を記録し、Implement Agentが63本のAPIと33画面を実装し、Test Agentが2,100+件のテストを書き、Review Agentが5回のレビューで重大なバグを発見し、Diary Agentが開発の軌跡を日記として残した。

この体制の核心は技術ではない。「役割を分けて、それぞれの専門性を高める」という、人間のチーム開発と同じ原則だ。AIエージェントを「全知全能の万能ツール」として使うのではなく、「専門性を持つチームメンバー」として配置する。その設計思想が、品質と生産性の両立を可能にした。

AI駆動開発はまだ始まったばかりだ。Claude Codeの進化とともに、サブエージェントの設計パターンも変わっていくだろう。しかし「役割の分離」「ドキュメントによるコンテキスト管理」「品質ゲートの仕組み化」という基本原則は、ツールが変わっても有効だ。

DevDexの開発記録が、あなたのAI駆動開発の一歩目になれば幸いだ。
