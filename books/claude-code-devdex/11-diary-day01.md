> 2026年3月11日（水）— コミット86件 / PR #1〜#81

## この日のゴール

「自分が使える用語管理ツールのMVPを1日で作る」。

朝8時にリポジトリを作り、CLAUDE.mdを書き、PM Agentにv0のタスク分解を依頼した。返ってきたのは14個のIssue。「1 issue = 1〜2時間」ルールの原型がここで生まれた。

## 朝〜昼: 基盤構築（#1〜#30）

最初の数時間で決めたことが、その後の全開発を支えることになる。

- **Next.js 16 + App Router + TypeScript** を選択。Server Actionsは使わない（将来のモバイル対応を見据えて、全API Route Handler方式に統一）
- **Supabase** でPostgreSQL + Auth + RLS。「自分のデータは自分だけ」をDB層で強制
- **shadcn/ui + Tailwind CSS v4** でUI。コンポーネントのカスタマイズ性を優先

DB設計で迷ったのは `term_relations`（関連用語）の設計。配列で持つか中間テーブルにするか。PM Agentに相談して中間テーブルに決めた。理由は「双方向の関連が扱いやすい」「将来 relation_type を追加できる」。この判断をCLAUDE.mdに書いたことで、以降のImplement Agentが同じ設計思想で実装してくれるようになった。

## 昼〜夕方: 用語CRUD + 認証（#31〜#55）

用語の登録・編集・削除・一覧・詳細。ここがDevDexの核だから、APIのパターンを徹底的に統一した。

```
checkAuth → バリデーション → DB操作 → { data, error } レスポンス
```

この「APIの型」を1本目のAPIで確立したことが大きかった。2本目以降はImplement Agentが「`/api/terms/route.ts` と同じパターンで」の一言で正しく実装してくれた。

認証はSupabase Authのメール+パスワード。ここは悩まなかった。

## 夕方〜夜: ダッシュボード + 検索 + ピン留め（#56〜#81）

ダッシュボードを作ったとき、初めて「自分のツール」として動くものが見えた。ピン留め用語が並び、カテゴリ別のレーダーチャートが描画される。まだデータは3件しかないけど、「これを埋めていきたい」という感覚が生まれた。

検索はiLIKEでの部分一致。シンプルだけど十分。

## この日の失敗

- **テストを後回しにした**。「まず動くものを」と思ってテストなしで進めたが、Day 2でTest Agentを動かしたとき、API 20本分のテストを一気に書くことになった。最初から1 API = 1テストのリズムで進めるべきだった。
- **UIに時間をかけすぎた**。shadcn/uiの調整に2時間近く使った。MVPでは機能優先でいい。
