## APIとは何か？

**API**（Application Programming Interface）は、フロントエンド（画面）とバックエンド（サーバー）の間で「注文」をやり取りする仕組みだ。

レストランで例えると:
- **フロントエンド** = お客さん（注文を出す）
- **API** = ウェイター（注文を厨房に伝え、料理を席に届ける）
- **バックエンド** = 厨房（注文に基づいて調理する）

お客さんが厨房に直接入らないように、ユーザーがデータベースに直接アクセスすることはない。必ずAPIを通してデータをやり取りする。

## REST APIの基本 — 4つの動詞

APIのリクエストには、「何をしたいか」を示す動詞がある。

| HTTPメソッド | 意味 | レストランで例えると |
|-------------|------|-------------------|
| **GET** | データを読む | メニューを見る |
| **POST** | データを作る | 注文する |
| **PUT / PATCH** | データを更新する | 注文を変更する |
| **DELETE** | データを削除する | 注文をキャンセルする |

たとえば「セッティング一覧を表示する」なら `GET /api/darts`、「新しいセッティングを登録する」なら `POST /api/darts` になる。

## Next.js のAPI Routes — ファイルを置くだけでAPIができる

Next.js では `app/api/` フォルダにファイルを置くだけでAPIが作れる。

```
app/api/
├── auth/[...nextauth]/route.ts   ← ログイン
├── dartslive-stats/route.ts      ← スタッツ取得
├── stripe/
│   ├── checkout/route.ts         ← 決済開始
│   └── webhook/route.ts          ← 決済通知受信
├── line/
│   ├── webhook/route.ts          ← LINE Bot受信
│   └── link/route.ts             ← アカウント連携
├── cron/
│   └── daily-stats/route.ts      ← 日次バッチ
└── admin/
    └── role/route.ts             ← ユーザーロール変更
```

フロントエンドとバックエンドが **同じプロジェクトに同居** するのが Next.js の大きなメリット。別途サーバーを立てる必要がない。

## ミドルウェア — 厨房に入る前のチェックポイント

すべてのAPIに共通する処理（認証チェック、エラーハンドリング）を毎回書くのは非効率だ。そこで **ミドルウェア** を使う。

ミドルウェアとは「本来の処理の前に自動で実行されるチェック機能」だ。レストランなら「厨房に入る前の衛生チェック」にあたる。

```typescript
// 3段階のミドルウェア
withAuth()       → ログインしているか？
withAdmin()      → 管理者か？（DB二重確認）
withPermission() → 特定の権限があるか？
```

組み合わせ例:

```typescript
// PRO限定のAPI
export const POST = withErrorHandler(
  withPermission(canUseDartslive, 'PRO限定です', handler),
  'dartslive-api'
);
```

この1行で「エラーハンドリング → 認証 → PRO権限チェック → 本処理」が自動的に実行される。

## レートリミット — 1分間60回まで

**レートリミット** は、同じユーザー（IPアドレス）が短時間に大量のリクエストを送ることを制限する仕組みだ。

なぜ必要か？
- サーバーへの過負荷を防ぐ
- 不正アクセス（ブルートフォース攻撃）を防ぐ
- スクレイピングによるデータの大量取得を防ぐ

darts Lab では **Upstash Redis** というサービスで「60リクエスト/分」のレートリミットを実装している。

## セキュリティの多層防御

APIのセキュリティは「1つの壁」ではなく「何重もの壁」で守る。

```
リクエスト受信
  → ① レートリミット（大量アクセスをブロック）
  → ② 認証チェック（ログインしているか？）
  → ③ 権限チェック（PROか？管理者か？）
  → ④ 入力バリデーション（zodでリクエスト内容を検証）
  → ⑤ 本処理（データの読み書き）
```

1つの壁が突破されても次の壁がある。これが **多層防御**（Defense in Depth）の考え方だ。

## まとめ

- APIは画面とサーバーの間の「ウェイター」
- REST APIは GET / POST / PUT / DELETE の4つの動詞
- Next.js ではファイルを置くだけでAPIが作れる
- ミドルウェアで認証・権限チェックを自動化
- レートリミットと多層防御でセキュリティを担保
